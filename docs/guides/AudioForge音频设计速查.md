# AudioForge 音频设计速查

> 这是面向音频设计与 TA 的快速入口，不替代完整说明。完整工作流、对象模型和交付边界统一以 `docs/guides/AudioForge使用说明.md` 为准。

当前文档同步日期：2026-05-14

适用对象：音频设计、音频 TA、负责日常内容生产和交付的同学。

## 1. 先记住这 6 句话

1. `.afproj` 是编辑源，`AudioData.json` 是运行时真源，不要反向手改导出 JSON 当主稿。
2. Event 负责触发规则，Audio Object 负责声音层配置，二者不要混着理解。
3. 先清 `Error` 再构建，`Warning` 不是都要立刻修，但必须知道风险是什么。
4. Event Id 要稳定，不要因为整理命名随手改掉已接入代码的事件。
5. 交付前至少跑一次全量构建，不要只靠选中构建做最终交包。
6. 听感确认看试听，交付确认看校验、构建和导出结果三者是否一致。

## 2. 每天最常用的工作流

### 2.1 打开工程

- 打开最新 `.afproj`。
- 检查自动恢复内容是否正确。
- 确认当前导出目录、运行时格式和默认 Bus 没有跑偏。

### 2.2 整理对象

- 用事件树组织业务语义。
- 用 Audio Object 承载声音层规则。
- 用源音频树管理素材，不把素材本身等同于事件。
- 用 Bus 管理混音层级，不把 Bus 当分类目录乱建。

### 2.3 导入音频

- 导到事件树：通常会创建 Event 和相关 Audio 关联。
- 导到 Audio 树：适合先整理声音对象。
- 导到源音频树：只入库，不自动进入运行时引用。

### 2.4 配参数

Event 重点看：

- `Event ID`
- `AudioId`
- `MaxInstances`
- `CooldownSeconds`
- `StealPolicy`

Audio Object 重点看：

- Bus
- PlayMode
- Clip 集合
- 音量 / 音高
- GameSync 绑定

Clip 重点看：

- 权重
- 资源键
- 导出路径
- `TrimStartMs`
- `TrimEndMs`
- `FadeInMs`
- `FadeOutMs`

### 2.5 听和看

- 先试听是否命中预期 Clip。
- 再看 Random / Sequence / Combo 行为是否符合预期。
- 最后看响度监视器的源读数和处理后读数。

### 2.6 校验和构建

- 先全量校验。
- 先修 `Error`。
- 日常迭代优先增量构建。
- 交付前至少跑一次全量构建。

## 3. 什么时候该用哪种播放模式

- OneShot：单次反馈，结构最简单。
- Random：多个候选随机命中，适合 UI 和轻变化反馈。
- Sequence：需要固定轮播顺序时使用。
- Combo：连续触发有步进语义时使用。

经验判断：如果你说不清为什么必须有“连续时间步进”，那大概率不该上 Combo。

## 4. 什么时候该用 GameSync

- RTPC：连续变化量，例如速度、强度、距离。
- State：全局离散状态，例如战斗、暂停、胜利。
- Switch：对象局部离散分支，例如材质、武器、角色类型。

如果只是普通分层或普通事件拆分，不要为了“系统感”硬上 GameSync。

## 5. 构建前 30 秒检查

- 当前对象有没有未保存修改。
- Event Id 有没有改动到研发已接入事件。
- Bus 路由是否正确。
- 有效 Clip 是否为空。
- 资源键和导出路径是否稳定。
- 问题中心里是否还有 `Error`。

## 6. 交付给研发时至少给什么

- 导出目录。
- 本轮版本说明。
- 关键 Event Id 列表。
- 如果这次升级了 SDK 或契约，明确提醒研发同步包和文档。

## 7. 最常见的 5 个坑

1. 改了 Event Id，却没同步告诉研发。
2. 试听正常，但忘了重新构建导出。
3. 只改了源音频文件，没有检查资源键和导出路径是否变化。
4. 用选中构建当最终交付，导致选区外脏资源没进包。
5. 把 Audio Object 和 Event 的职责重新混回一层，导致协作边界又乱了。
