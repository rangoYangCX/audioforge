# AudioForge Wwise 兼容工作台映射

## 目的

这份文档定义 AudioForge 下一阶段在本地桌面工具层的界面演进方向：

- 目标不是像素级复制 Wwise。
- 目标是复用 Wwise 用户最熟悉的工作台心智与任务路径。
- 范围只限本地 AudioForge 工具，不改 Unity 运行时输入边界。

## 不可变边界

本轮优化必须保持以下边界不变：

- 不修改 AudioData.json、AudioManifest.json、AudioEventID.cs、BuildReport.json 的既有字段语义。
- 不修改运行时音频资源目录结构和 Unity 侧读取路径。
- 不要求 Unity 侧读取 .afproj 或依赖 Python 运行环境。
- 不把编辑器内部布局状态、筛选状态或临时交互状态写进导出结果。

换句话说，这一轮只改“怎么编辑”，不改“导出给 Unity 什么”。

## 设计原则

### 1. 认知兼容优先于外观复制

团队成员熟悉的是：

- 左侧对象树
- 中央编辑工作台
- 右侧检查与上下文信息
- 底部日志、结果、分析入口

AudioForge 应该复刻这套工作流骨架，而不是复制 Wwise 的颜色、图标或具体控件外观。

### 2. 任务聚焦优先于控件堆叠

中央区域必须围绕当前任务组织：

- 事件设计
- 资源整理
- Bus 混音台
- 校验修复
- 构建交付

上下文信息改为收口到各工作区自身的上下文栏，不再固定占用右侧全局检视器，以避免窄窗口下的信息裁切和重复摘要。

### 3. 结果回看独立成坞

校验、构建、响度和日志需要始终有回看入口，但不应长期抢占主编辑空间。底部结果坞负责快速回看，完整分析仍保留结果中心。

## 当前推荐主布局

### 目标布局

- 左侧：工程浏览器
- 中央：当前工作区主编辑面
- 底部：结果坞
- 工作区内：按需显示的上下文栏

### 各区职责

左侧对象浏览器：

- 负责对象查找、层级组织、快速筛选和多选入口。
- 固定拆成三个分页：总线树在前，源音频树居中，事件树在后。
- 总线树页负责总线创建、子线创建、路由浏览和有效输出链展示。
- 源音频树页负责按编辑期路径浏览源文件、显示引用状态，并作为后续拖拽创建 Source Binding 的来源入口。
- 源音频树页还应提供状态过滤、定位引用事件、追加到当前事件等 editor-only 动作，不把这些交互状态写入导出结果。
- 事件树页负责 Folder / Event 浏览、事件类型图标识别和 Source Binding 悬浮窗入口；Source Binding 明细不再作为常驻树节点堆在事件下方。

中央工作台：

- 只承担当前任务的主要编辑动作。
- 事件模式下聚焦参数和试听。
- 资源模式下聚焦片段和批处理。
- 总线模式下聚焦路由和输出。

工作区上下文栏：

- 在事件、资源、总线、校验和构建页内按需显示任务导航。
- 需要时显示工程摘要、当前对象头和对象引用关系。
- 与主编辑区之间保持可手动拖拽缩放，不固定占用全局右侧栏。

底部结果坞：

- 常驻显示当前模式、当前结果焦点和脏状态。
- 一键跳到日志、校验、构建、响度结果。
- 不承担复杂编辑，只承担回看与定位。

## 本轮原型已落点

- 顶部全局搜索文案扩大到对象、总线、问题和结果。
- 顶部全局搜索执行时已支持跨对象跳转，可直接打开工程对象、总线和结果条目。
- 顶部命令按钮已收敛为导航与隐藏动作面板，负责工作区切换、结果跳转和少量未显式暴露的动作入口，不再重复顶栏与工作区里已可见的执行按钮。
- 取消固定右侧检视器，将工程摘要、当前对象和对象引用回收到各工作区上下文栏。
- 事件工作区不再重复展示对象头和引用卡片。
- Bus 混音台不再重复展示工程摘要卡片。
- Bus 混音台已增加路由图，可同时显示当前 Bus、父 Bus 链路、子 Bus 和有效输出。
- 资源工作区已增加最近批量反馈卡，批量权重、属性、重命名、排序和拖拽重排结果都会保留在当前事件上下文里。
- 底部结果区域从隐藏入口改为常驻结果坞。
- 底部结果坞已增加日志、校验、构建、响度四块结构化摘要，不再只有入口按钮。

## 二期对象浏览器设计

- 左侧入口升级为固定三分页：第一页为总线树，第二页为源音频树，第三页为事件树；三页共享搜索框与顶部上下文栏，但不共享节点类型。
- Bus 树页的最低闭环是“新建主线 / 新建子线 / 重命名 / 调整父子层级 / 查看有效输出链”。不要再把 Bus 设计拆回独立的临时面板或浮层。
- Bus 树页必须把 `当前 Bus -> 父 Bus -> ... -> Master` 的完整路由链显示出来，并同时给出 Effective Output Target、Effective Volume Db、Effective Mute 三个推导结果。
- 源音频树页的最低闭环是“按路径浏览 / 查看引用关系 / 标记缺失与未引用 / 从源定位到事件或资源页”。不要把源音频继续塞在事件树子节点里冒充层级对象。
- 事件设计页保留一个只读的源音频绑定摘要区：默认只显示当前事件挂了哪些源音频、各自的 Enabled / Active 状态和来源路径，不再承担主编辑入口。
- 事件列表页必须严格限制层级：Folder 下只能放 Folder 或 Event；Event 下只能放 Source Binding；事件不能成为事件的子节点。
- 源音频与 Event 解耦：从源列表或文件系统把源音频拖到 Event 上时，只创建 Source Binding，不复制文件、不新建 Event，也不改变源音频本体归属。
- 当前实现已落地：事件树默认只显示 Folder / Event，并按 OneShot、Random、Sequence、Combo 使用不同图标；展开事件节点会弹出 Source Binding 悬浮窗。
- 当前实现已落地：从源音频树拖到 Event 或事件树绑定悬浮窗，都会立即创建 binding 并给出成功/跳过重复项反馈；事件设计页旧绑定区只保留摘要回显，这些动作仍只作用于编辑器内的 `Event.clips`。
- Source Binding 需要区分 `Enabled` 与 `Active`：OneShot 只允许一个 `Active` 绑定参与运行时；Random / Sequence / Combo 可同时有多个 `Active` 绑定，并且新追加或重新启用的绑定默认会进入 Active 集合。
- 事件类型创建入口应直接暴露 OneShot、Random、Sequence、Combo，并在树中用不同图标区分；Sequence 额外显示是否启用 ShuffleBag，OneShot 额外显示是否启用 Loop。

## 对标结论

- 借鉴 Wwise：保留左侧对象浏览器常驻和总线路由可解释性，但不引入 SoundBank 或运行时对象分层。
- 借鉴 FMOD Studio：让分页切换与筛选服务于任务流，减少“对象树改一次、中央工作区跟着跳一次”的打断感。
- 借鉴 CRI ADX2：单独给源音频一棵树，承载素材索引、引用数和缺失检测；这些能力默认留在工具层，不改 Unity 运行时输入。
- 不在本轮引入 profiler 协议、远程联调桥和运行时回传结构，避免对当前 Unity SDK 开发造成额外变更。

## 下一轮建议

### 已完成 P1

- 给总线页增加路由图。
- 给资源页增加更完整的批量编辑反馈。

### P2

- 总线树 / 源音频树 / 事件树三分页对象浏览器落地，并补有效输出计算与源音频引用概览。
- 源音频树拖到事件树创建 Source Binding、事件树弹出 binding 悬浮窗，以及 Active / Enabled 状态联动落地。
- 事件类型创建入口、OneShot/Random/Sequence/Combo 图标体系与有效 binding 过滤逻辑落地。
- 事件页模式摘要卡和权重分布摘要后移评估，避免把过多状态继续暴露到主编辑面。

### P3

- 引入 Wwise 风格的 Game Sync authoring 范围：Project Explorer 需要能浏览并编辑 Game Parameters、State Groups、Switch Groups，而不是只停留在事件 / 资源 / Bus 三类对象。
- 事件与 Bus 的属性编辑器需要增加 Game Sync 绑定入口：RTPC 负责连续属性驱动，State 负责全局属性覆盖，Switch 负责离散分支选择；三者不能混成一个通用“参数区”。
- phase3 必须与 Unity runtime 设计联动，先确定 Schema v2、emitter 作用域和求值顺序，再决定工具端作者界面的最终形态。
- 工具层可以继续借鉴 Wwise 的 Game Sync 工作流，但不得为了 authoring 便利而篡改三类控制量的基础语义边界。

### P4

- 增加布局预设与自定义布局保存。
- 建立独立于 Wwise 的 AudioForge 设计语言与视觉资产。

## 三期说明

- 三期的优先级已经从“纯工作台体验深化”切换为“Wwise Game Sync 语义适配”：核心问题不再是左侧树怎么摆，而是 RTPC / State / Switch 如何落进 AudioForge 工程模型、导出契约和 Unity runtime。
- phase3 不是对 phase2 编辑器的附属小补丁，而是当前仓库第一次明确允许升级运行时契约的阶段；相关详细规划以 `docs/internal/audioforge_第三期RTPC-State-Switch实施计划.md` 和 `docs/UnityRuntime三期GameSync设计.md` 为准。
