# AudioForge 桌面工具产品化路线图

当前文档同步日期：2026-05-14

> 当前状态：本文件已接替原 `audioforge_商业化改造清单.md` 与 `audioforge_wwise兼容工作台映射.md`，作为桌面工具产品化方向的唯一主文档。原文档已移入 `docs/internal/archive/`。

## 1. 当前判断

桌面工具的第一轮产品化重构已经完成：

- AppShell、顶部应用栏、左侧任务导航、欢迎页和结果中心已落地。
- 主布局已收敛为“左侧对象浏览器 + 中央工作台 + 底部结果坞”。
- phase3 的 GameSync、Audio Object 主模型和 Wwise 风格工作区心智已经进入主线。

当前的主要差距不再是“有没有功能”，而是三类能力还需要继续收口：

- 任务型信息架构是否足够稳定。
- 高频编辑控件是否足够专业和可扩展。
- 诊断、交付与批处理是否足够像成熟音频工具。

## 2. 不可变边界

这一轮桌面工具产品化必须保持以下边界不变：

- 不修改 `AudioData.json`、`AudioManifest.json`、`AudioEventID.cs`、`BuildReport.json` 的既有字段语义。
- 不修改 Unity 侧运行时输入结构与资源读取边界。
- 不要求 Unity 侧读取 `.afproj` 或依赖 Python 运行环境。
- 不把编辑器内部布局状态、筛选状态或临时交互状态写入导出结果。

换句话说：这一轮优化只改“怎么编辑”，不改“给 Unity 什么”。

## 3. 设计原则

### 3.1 认知兼容优先于外观复制

目标不是复制 Wwise 的外观，而是复用熟悉中间件用户的工作台心智：

- 左侧对象浏览器
- 中央任务工作台
- 底部结果回看
- 按工作区出现的上下文栏

### 3.2 任务优先于控件堆叠

中央区域必须始终围绕当前主任务组织：

- 事件设计
- 资源整理
- Bus 混音台
- 校验修复
- 构建交付

### 3.3 结果回看独立成坞

日志、校验、构建和响度必须始终可回看，但不应长期抢占主编辑区。复杂分析继续留在结果中心，日常回看留在底部结果坞。

### 3.4 主文档单一化

产品方向只维护这一份路线图，不再让“差距清单”“映射文档”“页面蓝图”并行演进。

## 4. 当前工作台映射

| 中间件心智 | AudioForge 当前落点 | 后续重点 |
| --- | --- | --- |
| Project Explorer | 左侧对象浏览器 | 强化筛选、收藏、最近编辑 |
| Event / Property Editor | 中央视图中的事件设计页 | 基础/高级分层、批量属性桥接 |
| Contents Editor | 资源整理页与片段编排区 | model/view 化、排序与局部刷新 |
| Bus Mixer | Bus 混音台与路由图 | 更清晰的有效输出链和快照感知 |
| Capture / Errors | 结果坞与结果中心 | 结构化详情、问题聚合、联调入口 |
| Search / Command | 顶部全局搜索与命令入口 | 搜索结果分层、跨区动作融合 |

## 5. P0 持续收口项

### 5.1 信息架构稳定化

- 保持“左树 + 中工作台 + 工作区上下文栏 + 底部结果坞”作为默认框架。
- 减少重复说明卡、重复操作入口和并行状态摘要。
- 让新用户在 10 秒内判断“当前页面正在做什么”。

### 5.2 高频控件专业化

- 事件树、Audio 树、源音频树继续向更稳定的 model/view 结构演进。
- 片段表进一步支持排序、筛选、局部刷新和批量反馈。
- 大工程下优先保证浏览、筛选、批量操作不卡顿。

### 5.3 反馈系统产品化

- 保存、校验、构建、导出要有明确结果反馈。
- 问题中心继续强化“对象定位 + 原因 + 下一步建议”。
- 构建差异和发布结果优先用结构化摘要表达，而不是纯文本堆叠。

### 5.4 首次使用体验

- 欢迎页继续承担最近工程、快速开始和文档入口。
- 空状态继续保持任务导向，不回退为禁用态堆控件。

## 6. P1 方向

### 6.1 诊断与交付中心增强

- 结果中心进一步向“诊断与交付中心”演进。
- 统一事件、Bus、构建差异、验证签收与联调入口。

### 6.2 模板与批处理资产化

- 引入常用事件模板、Bus 指派模板和参数预设。
- 强化“当前对象与模板差异”的可视化反馈。

### 6.3 搜索与命令融合

- 顶部搜索结果按对象、问题、结果分层。
- 命令面板继续只承载导航和隐藏动作，不回退为第二套主操作区。
