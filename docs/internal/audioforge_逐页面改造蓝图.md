# AudioForge 逐页面改造蓝图

## 文档目标

这份蓝图用于把前面的产品判断进一步落到页面层，直接回答三个问题：

1. 当前页面各自的问题是什么。
2. 商业软件级版本里，这个页面应该长成什么样。
3. 实施时应该按什么顺序拆解，避免重构失控。

这份文档默认基于当前主窗口结构展开，主要对应以下现状：

- 单一主工具栏承载大量动作。
- 主编辑区由对象头、属性页、内容页、响度页组成。
- 底部常驻日志与报告中心。
- 项目、事件、音频属性、生成、总线工作区都嵌在同一个主窗口构建函数中。

## 总体重构原则

### 1. 页面优先级高于控件优先级

先定义页面，再定义每个页面里有哪些控件，而不是继续从现有控件拼接页面。

### 2. 每个页面只服务一个主任务

如果一个页面同时承载“编辑”“分析”“报告”“修复”“交付”四种目标，就一定会过载。

### 3. 全局动作收口到应用栏

工程级动作和编辑级动作分离。页面内部只保留与当前任务直接相关的动作。

### 4. 页面的信息层级必须固定

每个页面都遵循统一顺序：

- 页头摘要
- 主工作区
- 辅助上下文
- 结果反馈

### 5. 报告中心从“常驻”改成“按需展开”

日志和长文本报告不再与主编辑区争空间。

## 页面 0：应用壳层

## 当前问题

- 现在的主工具栏同时承担工程、对象、修订、试听、校验、构建、视图、设置。[audioforge/app/views/main_window.py](audioforge/app/views/main_window.py#L483)
- 顶部操作密度过高，缺少明确的主按钮和次级按钮层级。
- 用户打开软件后，首先看到的是“能做什么”，而不是“现在最该做什么”。

## 目标状态

应用壳层改成三段：

- 顶部应用栏
- 左侧任务导航
- 中央页面容器

### 顶部应用栏

只保留全局动作：

- 项目名称
- 项目路径与保存状态
- 全局搜索
- 命令面板入口
- 主构建按钮
- 设置入口

不再在顶部塞入新建事件、批量改总线、缩放、停总线这类页面级动作。

### 左侧任务导航

固定为 5 个一级入口：

- 资源整理
- 事件设计
- 总线与混音
- 校验修复
- 构建交付

每个入口允许有自己的二级页签，但不允许在顶层横向堆太多 Tab。

## 页面级实施项

- 把现有 toolbar section label 模式拆掉。
- 提取 AppShell 组件，管理顶部栏、左导航、页面切换。
- 现有主窗口只保留宿主职责，不再继续扩张。

## 页面 1：欢迎页

## 当前问题

- 当前没有真正的欢迎页，用户进入软件后直接面对完整编辑器。
- 对新用户来说，入口太重。

## 目标状态

欢迎页承担商业软件的第一印象。

布局建议：

- 左侧：最近工程
- 中央：新建工程、打开工程、示例工程
- 右侧：快速开始、最近更新、文档入口

必须提供：

- 空工程模板
- 休闲音效模板
- UI 事件模板
- BGM 模板

## 页面级实施项

- 新增 WelcomePage 组件。
- 引入最近工程数据源。
- 为首次启动单独设计空状态文案与视觉卡片。

## 页面 2：资源整理

## 当前问题

- 当前“内容编辑器”里把片段列表、片段详情、批处理、生成预览塞在一起。[audioforge/app/views/main_window.py](audioforge/app/views/main_window.py#L805)
- 用户在整理素材时，会被导出预览、批处理建议、单片段详情同时打扰。
- 片段表仍然偏内部工具式表格，不够专业。

## 目标状态

资源整理页只负责素材进入系统后的标准化工作：

- 导入
- 命名
- 资源键清洗
- 标签整理
- 批量归类

布局建议：

- 左侧：素材来源与筛选器
- 中央：片段表
- 右侧：片段详情 / 批量修改面板

### 左侧区

- 当前事件所属素材
- 未绑定素材
- 最近导入素材
- 标签筛选
- 来源筛选

### 中央视图

- 专业表格
- 可配置列
- 排序
- 状态徽标
- 行内错误提示

### 右侧区

切换两种模式：

- 单片段详情
- 多选批量编辑

## 页面级实施项

- 将现有片段列表与片段详情从 contents_tabs 中抽离成独立页面。
- 批处理不再做成单独 Tab，而是作为右侧模式切换。
- 生成预览从本页移走，避免任务污染。

## 页面 3：事件设计

## 当前问题

- 当前属性编辑器里，事件、音频属性、生成、工程都混在 property_tabs 里。[audioforge/app/views/main_window.py](audioforge/app/views/main_window.py#L786)
- 对象头、对象引用、属性页签叠加后，信息密度过大。
- “事件设计”这一最核心工作流，没有获得绝对焦点。

## 目标状态

事件设计页应该是整套产品最强的一页。

布局建议：

- 左侧：事件树
- 中央：事件属性编辑区
- 右侧：事件摘要与片段编排

### 左侧事件树

必须强化：

- 搜索
- 多选
- 右键命令
- 状态徽标
- 校验级别颜色

### 中央事件属性区

分成 4 个固定区块：

- 基础信息
- 播放行为
- 增益与音高
- 高级控制

Combo 相关参数仅在 Combo 模式展开，不占默认空间。

### 右侧事件摘要区

只显示用户最常需要扫一眼的信息：

- 当前模式
- 所属总线
- 片段数
- 权重分布
- 冷却/实例上限
- 最近试听结果
- 当前问题数

并提供两个固定动作：

- 试听
- 跳到问题

## 页面级实施项

- 事件页从 property_tabs 独立出来，成为一级页面。
- 对象头重做成更精简的 Context Summary Bar。
- 对象引用区不再常驻完整表格，而是收纳到可展开详情。

## 页面 4：总线与混音

## 当前问题

- 当前总线工作区已经开始成型，但仍以表单和标签为主。[audioforge/app/views/main_window.py](audioforge/app/views/main_window.py#L620)
- 路由关系、有效输出和总线结构没有形成真正的图形化理解入口。
- Master、总线浏览器、总线工作区之间的视觉组织仍然偏“工程配置页”。

## 目标状态

总线与混音页是理解整个项目增益结构的地方。

布局建议：

- 左侧：总线树
- 中央：总线路由图 + 输出链图
- 右侧：当前总线属性与试听区

### 左侧总线树

- 层级关系
- 默认总线标记
- 静音标记
- 导出状态标记

### 中央视图

至少包含：

- 当前总线到 Master 的路由链
- 各级总线增益叠加关系
- 静音阻断点
- 下游总线预览

### 右侧属性区

- 基础音量
- 静音
- 父总线
- 默认状态
- 最近试听影响
- 导出结果说明

## 页面级实施项

- 把当前 inline bus 和 project bus 两套视图统一成一个 Bus Workspace。
- 用可视化组件替换 route bar 的文本式表达。
- Master 从单独表单升级成整页中的核心锚点。

## 页面 5：校验修复

## 当前问题

- 当前校验报告是 report_tabs 里的一部分，视觉上仍像日志附属页。[audioforge/app/views/main_window.py](audioforge/app/views/main_window.py#L898)
- 用户修问题时仍要和日志、构建报告、响度扫描共享空间。
- 问题中心虽然已有基础，但还没有成为完整的修复工作台。

## 目标状态

校验修复页应当是“问题收敛中心”。

布局建议：

- 左侧：问题筛选器
- 中央：问题列表
- 右侧：问题详情与修复动作

### 左侧筛选器

- 严重程度
- 对象类型
- 所属总线
- 问题来源
- 是否已处理

### 中央列表

- 问题标题
- 对象名称
- 严重程度
- 风险类型
- 最后更新时间

### 右侧详情

- 详细描述
- 为什么这是问题
- 对导出的影响
- 一键跳转对象
- 建议修复动作

## 页面级实施项

- 从 report_tabs 中拆出独立页面。
- 校验报告文本框降级为高级详情，不再作为主反馈。
- 增加“问题数总览卡片”和“修复进度”。

## 页面 6：构建交付

## 当前问题

- 当前生成设置和生成概览仍在 property_tabs 中；生成预览又在 contents_tabs 中；构建报告还在 report_tabs 中。[audioforge/app/views/main_window.py](audioforge/app/views/main_window.py#L713)
- 同一条构建工作流被拆散在三个区域。
- 用户无法在一个地方完成从配置到差异预览到交付结果的闭环。

## 目标状态

构建交付页应该像发布中心。

布局建议：

- 左侧：构建配置
- 中央：交付摘要和差异预览
- 右侧：构建结果和产物定位

### 左侧配置区

- 导出目录
- 源格式
- 运行时格式
- 构建选项
- 交付说明模板

### 中央摘要区

- 本次构建将输出哪些文件
- 和上次构建相比哪些对象发生变化
- 风险提醒
- 快速验收摘要

### 右侧结果区

- 构建状态
- 构建日志摘要
- 产物路径
- 一键打开导出目录
- 一键复制交付摘要

## 页面级实施项

- 生成设置、生成预览、构建报告三处内容统一收口到一个 Build Center。
- 文本预览改成结构化 diff 列表。
- 构建结果做成摘要卡片，不再只保留文本输出。

## 页面 7：结果中心

## 当前问题

- 当前 log_panel 常驻底部，占据持续空间。[audioforge/app/views/main_window.py](audioforge/app/views/main_window.py#L907)
- 对普通使用来说，这个区域大部分时间价值不高，却长期存在。

## 目标状态

结果中心改成可折叠抽屉或右侧弹出面板。

它只在以下场景出现：

- 构建完成后
- 校验完成后
- 响度扫描完成后
- 用户主动打开日志时

### 内部结构

- 顶部：结果摘要卡片
- 中部：结果类别切换
- 底部：详细文本和原始日志

## 页面级实施项

- log_panel 不再常驻主布局。
- report_tabs 改造成 Results Drawer。
- 工作区状态栏只保留最关键摘要，不承载完整结果入口。

## 页面 8：设置中心

## 当前问题

- 当前设置更像一个工具栏入口，不是清晰的系统中心。
- UI 缩放、布局恢复、路径偏好、主题偏好未来都会继续增长。

## 目标状态

设置中心作为独立对话框或独立页，至少覆盖：

- 通用
- 布局与外观
- 导出偏好
- 验证偏好
- 快捷键与效率

## 页面级实施项

- 把未来高频扩展项都从主工具栏移出，收口到设置中心。
- 为后续主题系统和命令面板预留入口。

## 模块拆分建议

为了支持以上页面重构，建议把当前主窗口拆成以下 UI 模块：

- AppShell
- TopAppBar
- TaskSidebar
- ContextSummaryBar
- ResourceWorkspace
- EventWorkspace
- BusWorkspace
- ValidationWorkspace
- BuildWorkspace
- ResultsDrawer
- WelcomePage

当前 main_window.py 更适合变成装配层，而不是继续承载全部页面细节。

## 实施顺序

### 第一阶段

- 新建 AppShell
- 新建 WelcomePage
- 把结果中心抽屉化
- 把顶部应用栏和左导航先搭起来

### 第二阶段

- 先迁移事件设计页
- 再迁移构建交付页
- 然后迁移校验修复页

### 第三阶段

- 重做资源整理页
- 重做总线与混音页
- 补齐设置中心

## 最终目标

页面级重构完成后，AudioForge 应该具备以下感受：

- 用户打开软件后，先看到任务入口，而不是控件堆。
- 每个页面只服务一个主任务，不再互相干扰。
- 报告和日志不再成为主界面负担。
- 事件设计和总线混音两个核心页面真正拥有专业工具感。
- 产品开始具备商业软件级的信息架构和长期演化能力。
