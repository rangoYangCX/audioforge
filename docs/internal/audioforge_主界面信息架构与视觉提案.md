# AudioForge 主界面信息架构与视觉提案

> 当前状态：2026-04-30 本提案中的 AppShell、顶部应用栏、左侧任务导航、欢迎页、结果中心和主工作区优先级重排已完成第一轮落地。本文档保留为视觉与信息架构依据，未注明完成的条目应视为后续继续打磨项。

## 目标

这份提案用于回答两个问题：

1. 新主界面应该如何组织，才能更像商业软件。
2. AudioForge 的视觉方向应该如何形成自己的产品识别，而不是继续停留在“深色专业工具”的通用外观。

## 当前问题总结

当前主界面的优势：

- 功能覆盖广。
- 工程树、属性、内容、报告、响度都已具备基础能力。
- 高频操作入口已经很多。

当前主界面的主要问题：

- 顶层结构偏控件导向，不是任务导向。
- 工具栏承载过多动作，视觉优先级不清楚。
- 属性页、内容页、报告页、日志区同时争夺注意力。
- 用户需要先理解“有哪些区域”，才能开始工作。
- 视觉上更像成熟内部工具，不像品牌化产品。

## 新主界面原则

### 1. 任务优先

用户进入软件后，第一眼应该看到当前最重要任务，而不是所有功能入口。

### 2. 对象上下文持续可见

当前选中的 Event、Bus、Clip、项目状态必须持续存在于界面头部，但摘要要精简。

### 3. 编辑区只保留一个主焦点

任何时刻，界面里只允许一个区域是“主工作区”，其他区域只能做辅助，不应争夺视线。

### 4. 报告和日志降级为结果中心

大部分时间不应常驻占空间，需要时再展开。

### 5. 可视化成为一等公民

响度、总线、导出差异、问题分布必须优先使用图形摘要，而不是文本堆积。

## 新主界面结构

## 顶层导航

建议从当前单工具栏，改为三层结构：

- 顶部应用栏
- 左侧任务导航
- 中央主工作区

### 顶部应用栏

承载全局级动作：

- 项目名称与路径
- 保存状态
- 全局搜索
- 快速命令入口
- 构建与导出主按钮
- 设置

这里不再承载所有编辑动作。

### 左侧任务导航

建议固定为 5 个一级入口：

- 资源整理
- 事件设计
- 总线与混音
- 校验修复
- 构建交付

它们代表用户的任务阶段，而不是控件区域。

## 中央主工作区

### 1. 资源整理

目标：导入素材、清洗命名、建立资源键、批量标签化。

布局建议：

- 左侧：素材树 / 文件导入区
- 中央：片段表
- 右侧：片段详情 / 批量编辑面板

该模式下不显示复杂事件参数。

### 2. 事件设计

目标：编辑 Event 本身。

布局建议：

- 左侧：事件树
- 中央：事件属性区
- 右侧：当前事件摘要、片段编排、试听摘要

这里是产品的核心工作区，应该拥有最高视觉优先级。

### 3. 总线与混音

目标：查看总线结构、调整基线、理解输出路径。

布局建议：

- 左侧：总线树
- 中央：总线路由图 + 音量结构
- 右侧：当前总线属性、父子关系、导出结果和试听信息

这里应突出图形化路由，而不是把总线区做成纯表单。

### 4. 校验修复

目标：集中处理问题，不被其他编辑任务干扰。

布局建议：

- 左侧：问题筛选器
- 中央：问题列表
- 右侧：问题详情与一键跳转

这里不应该只是日志面板换皮，而应该是专门的问题工作台。

### 5. 构建交付

目标：准备导出、查看差异、输出交付内容。

布局建议：

- 左侧：构建配置
- 中央：交付摘要与差异预览
- 右侧：构建结果、产物定位、签收摘要

这里要像发布中心，而不是当前的几个按钮加报告文本框。

## 对象头部重构建议

保留对象头，但要更克制。

建议只保留：

- 对象类型
- 对象名称
- 所属总线
- 当前模式
- 片段数
- 问题数
- 两个主动作：试听、定位问题

把次级信息例如路径、来源、生成方式，收纳到可展开的“对象详情”区。

## 报告中心重构建议

当前日志、校验报告、构建报告、响度扫描并排存在，像内部控制台。

建议改成统一的结果中心：

- 顶部：摘要卡片
- 中部：按类型分组的问题流
- 底部：详情区

只有高级用户才需要完整日志全文。

## 视觉方向

## 核心关键词

建议视觉关键词为：

- 精确
- 克制
- 稳定
- 专业
- 轻量

这意味着：

- 不走夸张炫技路线
- 不走廉价后台风
- 不照搬 Wwise/FMOD 的既有气质

## 配色方向

建议继续使用深色基底，但不要停在常见灰蓝工具风。

推荐方向：

- 基础背景：偏中性石墨灰
- 结构层：更明显的分层灰度
- 品牌强调色：偏暖的铜橙或偏冷的青金，只选一种
- 成功状态：干净明亮但不荧光
- 警告状态：更克制，不用大面积高饱和黄

视觉目标：

- 一眼看上去像“专业创作工具”
- 不是普通后台系统
- 不是游戏风 UI

## 字体与层级

建议建立明确层级：

- 应用标题
- 工作区标题
- 面板标题
- 表单标签
- 摘要正文
- 次要说明

现在很多信息都以相似权重出现，导致视觉噪音过大。

## 图标方向

建议图标统一成一套细线条 + 轻填充风格，避免不同页面的图标语义不一致。

重点不是数量，而是：

- 构建
- 校验
- 试听
- 总线
- 事件
- 资源
- 问题

这些核心对象要一眼可识别。

## 可视化方向

### 响度

- 折线图 + 峰值点 + 参考线
- 支持 source / processed 对比
- 支持 hover 读值

### 总线

- 树形路由图
- 输出链增益可视化
- 静音与默认总线状态徽标

### 事件

- 事件卡片摘要
- 当前模式与素材数可视化
- 权重分布条

### 构建

- 产物结构卡片
- 差异预览
- 风险提醒

## 首页提案

建议增加欢迎首页：

- 最近工程
- 新建工程
- 打开工程
- 示例工程
- 快速开始
- 最近版本更新

商业软件需要一个稳定的首页作为产品门面，而不是直接把用户扔进一堆控件。

## 推荐实施顺序

### 第一轮

- 先做顶层信息架构
- 再做设计 token
- 最后迁移主窗口

### 第二轮

- 重做事件设计工作区
- 重做总线工作区
- 重做校验与构建中心

### 第三轮

- 增加首页
- 增加命令面板与全局搜索
- 增加强化可视化

## 最终目标

理想状态下，AudioForge 的主界面应该让用户感受到：

- 这是一个成熟的专业音频工具
- 任务路径清楚，不需要先理解太多技术结构
- 重要信息会主动浮现，不需要到处翻日志
- 视觉上克制但有识别度
- 看起来不是 Wwise 的影子，而是 AudioForge 自己
