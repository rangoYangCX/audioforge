# AudioForge 日志与诊断暴露清单

当前文档同步日期：2026-05-14

当前适用版本：AudioForge 0.09.1

## 1. 文档定位

这份文档只回答一件事：AudioForge 的日志与诊断信息，应该按哪些暴露端口分类对外提供，以及每个端口至少要暴露哪些内容。

这里的“端口”不是网络端口，而是日志和诊断信息对外可见的出口，包括：

- 桌面工具 UI。
- 桌面工具日志文件。
- 构建报告与导出报告。
- Unity 运行时调试 API。
- Unity 调试面板 / Inspector。
- 自动化验证与发布签收材料。

这份清单的用途有三个：

1. 帮工具端确定哪些信息必须落日志，哪些应该直接露给用户界面。
2. 帮 Unity 运行时确定哪些调试字段必须可读，避免只能靠猜。
3. 帮后续做结果中心、调试面板、CI 报告和对外交付时，有一份统一字段基线。

## 2. 总原则

### 2.1 同一件事，至少要有一个稳定出口

关键诊断信息不能只停留在临时 UI 提示里，也不能只埋在底层文件日志中。每类关键问题至少要有一个稳定、可复查的出口。

### 2.2 结果、原因、上下文要一起暴露

只给“失败了”没有价值。至少要同时给：

- 发生了什么。
- 发生在哪个对象或资源上。
- 为什么失败。
- 当时的关键上下文。

### 2.3 面向不同消费者给不同粒度

- 音频设计更需要摘要、对象定位和可执行提示。
- 工具研发更需要结构化字段和详细错误链路。
- Unity 研发更需要事件、总线、资源和 GameSync 的运行时求值结果。
- 自动化链路更需要稳定的 JSON、计数和状态码。

## 3. 所有端口都应共享的基础字段

无论最终暴露到哪个端口，以下字段都应尽量统一：

- `timestamp`
- `level`
- `subsystem`
- `message`
- `session_id` 或本次运行标识
- `project_name`
- `project_path`
- `schema_version`
- `tool_version` 或 `sdk_version`
- `export_root`
- `correlation_id`

推荐补充：

- `thread_name`
- `pid`
- `platform`
- `error_code`
- `failure_stage`
- `elapsed_ms`

## 4. 端口一：桌面工具 UI

消费方：音频设计、音频 TA、工具使用者。

目标：不要求用户先翻日志文件，也能定位大部分可修问题。

### 4.1 必须暴露

- 当前日志路径，例如 `latest.log` 和 `latest.fault.log` 的实际位置。
- 最近一次构建状态。
- 最近一次校验状态。
- 当前工程是否存在未保存修改。
- 当前构建请求范围，例如全量、增量、选中构建。
- 当前构建目标，例如 `selection_label`。
- 当前导出目录。
- 校验错误数、警告数、信息数。
- 构建成功后的关键产物路径。
- 构建失败时的直接原因。
- 构建被校验阻断时的错误数和阻断说明。
- 最近一条关键日志摘要。
- 关键对象跳转入口，例如从问题中心跳回 Event、Audio Object 或 Bus。

### 4.2 建议暴露

- 最近失败资源条目。
- 最近失败对象条目。
- 构建计划摘要。
- `rebuilt_asset_count`。
- `reused_asset_count`。
- `removed_asset_count`。
- `out_of_scope_dirty_count`。
- 最近一次响度扫描摘要。
- 最近一次试听失败摘要。

### 4.3 更适合以 UI 摘要呈现的内容

- “当前构建失败，原因是导出父目录被文件占用”。
- “当前事件无有效 Clip，构建已阻断”。
- “诊断日志路径已就绪，可直接打开”。
- “当前请求被忽略，因为已有构建任务正在执行”。

## 5. 端口二：桌面工具持久日志文件

消费方：工具研发、问题复现人员、发布验证人员。

目标：把一次运行中的关键行为写成可复查的时序记录。

当前建议至少包括以下文件：

- `session_log`
- `latest.log`
- `latest.fault.log`

### 5.1 必须暴露

- 应用启动记录，包含会话日志路径、latest 路径、fault 路径。
- 工程打开、保存、另存为、自动恢复行为。
- 构建请求开始记录。
- 构建请求范围、`selection_label`、`export_root`。
- 构建计划摘要和关键计数。
- 导出开始。
- 导出提交成功。
- 导出失败。
- 每个资源导出的开始记录。
- 每个资源导出的成功记录。
- 每个资源导出的失败记录。
- 资源级日志中的 `asset_key`。
- 资源级日志中的 `source_path`。
- 资源级日志中的 `target_path`。
- 资源级日志中的 `index / total`。
- 资源级日志中的 `elapsed_ms`。
- 复用资源缺失并回退重建的告警。
- 未捕获异常。
- 线程异常。
- unraisable exception。
- Qt warning / critical / fatal 信息。

### 5.2 建议暴露

- 当前工程文件路径。
- 当前导出目录的解析前后值。
- 构建线程开始和清理完成。
- 校验阻断摘要。
- 关键服务切换或降级路径，例如重编码回退、复制直通等策略变化。
- 响度扫描开始与结束。
- 试听初始化、设备异常和回退路径。

### 5.3 资源级日志最少字段模板

- `action`
- `asset_key`
- `source_path`
- `target_path`
- `index`
- `total`
- `file_size`
- `elapsed_ms`
- `result`
- `exception`

## 6. 端口三：构建报告与导出报告

消费方：音频设计、工具研发、CI、交付验收人员。

目标：把“这次构建到底做了什么”固化成稳定报告，而不是只看界面结果。

### 6.1 必须暴露

- `requested_scope`
- `effective_scope`
- `selection_label`
- `rebuilt_asset_keys`
- `reused_asset_keys`
- `removed_asset_keys`
- `out_of_scope_dirty_asset_keys`
- `export_root`
- `data_file`
- `manifest_file`
- `report_file`
- 构建是否成功。
- 构建失败时的错误原因。

### 6.2 建议暴露

- 构建开始时间和结束时间。
- 总资源数。
- 本次新增资源数。
- 本次移除资源数。
- 本次受影响的 Event 数。
- 本次受影响的 Audio Object 数。
- 校验摘要。
- 关键警告摘要。

### 6.3 这类端口更适合机器消费

- `BuildReport.json`
- `full_chain_report.json`
- 发布签收 JSON

### 6.4 这类端口更适合人读

- `BuildReport` 的摘要文本。
- 全链路报告 Markdown。
- 发布签收 Markdown。

## 7. 端口四：Unity 运行时调试 API

消费方：Unity 客户端研发、运行时研发、验证脚本。

目标：不给 Unity 研发留下“只能听有没有声音，不能看运行时到底做了什么”的黑盒。

### 7.1 必须暴露

- 初始化是否完成。
- 事件总数。
- 总线总数。
- 当前运行时音频格式。
- 当前资源提供器类型。
- 当前处理后 Clip 缓存数量。
- 最近事件触发记录查询能力。
- 最近总线状态变化记录查询能力。
- 当前 Event Id 列表查询能力。
- 当前 Bus 名称列表查询能力。
- 当前 Event 活跃声部数量查询能力。
- 当前 Bus 音量查询能力。
- 当前 Bus 静音状态查询能力。

### 7.2 最近事件记录至少要包含

- `EventId`
- `ClipId`
- `AssetKey`
- `BusName`
- `Result`
- `Message`
- `VolumeDb`
- `PitchCents`
- `ComboStep`
- `UsedTimePreservingPitch`
- `CacheHit`
- `TrimStartMs`
- `TrimEndMs`
- `FadeInMs`
- `FadeOutMs`
- `LoopStartMs`
- `LoopEndMs`
- `Timestamp`

### 7.3 最近总线记录至少要包含

- `BusName`
- `Action`
- `Volume`
- `IsMuted`
- `Timestamp`

### 7.4 建议暴露

- 事件基础音量查询。
- 事件当前 Bus 查询。
- 当前全局 State 快照。
- 当前全局 Game Parameter 快照。
- 指定 emitter 的本地 Game Parameter 快照。
- 指定 emitter 的 Switch 快照。
- 最近一次资源加载失败原因。
- 最近一次播放被拒绝原因。

## 8. 端口五：Unity 调试面板 / Inspector

消费方：Unity 联调人员、音频设计、TA。

目标：把 API 能读到的关键信息变成低门槛可视化入口。

### 8.1 必须暴露

- Runtime Ready 状态。
- 当前事件数。
- 当前总线数。
- 当前运行时音频格式。
- 当前资源提供器类型。
- 是否启用保时长变调。
- 当前处理后 Clip 缓存数量。
- 当前 Bus 状态列表。
- 当前 Event 活跃声部列表。
- 最近事件记录列表。
- 最近总线记录列表。

### 8.2 建议暴露

- 最近一次资源加载失败。
- 最近一次初始化失败。
- 最近一次 GameSync 求值结果变化。
- 当前全局 State 摘要。
- 当前全局 RTPC 摘要。
- 当前 emitter 作用域摘要。
- 运行中切换显示条数和过滤条件。

### 8.3 更适合面板直接显示的摘要句

- “AudioData.json 未找到”。
- “事件已命中，但被冷却拦截”。
- “事件已命中，Clip 为空，未播放”。
- “当前使用 StreamingAssetsProvider”。
- “当前命中保时长变调缓存”。

## 9. 端口六：自动化验证、发布签收与对外交付材料

消费方：发布负责人、CI、项目组、外部接收研发。

目标：让交付包不仅能用，还能证明“这次是怎么验过的”。

### 9.1 必须暴露

- 当前版本号。
- 当前文档同步日期。
- 当前验证命令。
- 验证结果摘要。
- `pytest` 通过情况。
- 烟雾工程验证结果。
- 全链路检查结果。
- 当前签收结论。
- 当前报告文件路径或目录。

### 9.2 建议暴露

- 关键变更摘要。
- 当前已知风险。
- 当前未覆盖项。
- 当前平台边界，例如 mac 仍需真机确认项。
- 当前包内 canonical 文档列表。

## 10. 按消费者反推的最小暴露集合

### 10.1 音频设计最少要看到什么

- 当前对象是否有错误。
- 当前构建是否成功。
- 当前导出目录在哪里。
- 最近失败资源是什么。
- 最近关键日志摘要是什么。

### 10.2 工具研发最少要看到什么

- 完整时序日志。
- 资源级开始 / 成功 / 失败记录。
- 崩溃和线程异常。
- Qt 消息。
- 导出提交和回滚链路。

### 10.3 Unity 研发最少要看到什么

- 初始化结果。
- Event / Audio Object / Bus 的解析结果。
- 最近事件触发记录。
- 最近总线变化记录。
- 资源加载和播放拒绝原因。

### 10.4 发布与 CI 最少要看到什么

- 验证命令。
- 验证结果。
- 报告文件。
- 版本号和签收结论。

## 11. 当前最值得优先补齐的缺口

如果只按排障收益排序，建议优先补这几类暴露：

1. Unity 运行时的资源加载失败原因统一结构化输出。
2. 播放被拒绝原因统一结构化输出，例如冷却、实例上限、无有效 Clip、GameSync 未命中。
3. 桌面工具 UI 直接显示最近失败资源和最近失败对象，而不是只显示一条总括状态。
4. 构建报告补统一开始时间、结束时间、耗时和对象影响面统计。
5. 调试面板补全当前全局 State / RTPC / emitter 上下文摘要。

## 12. 推荐落地顺序

1. 先统一基础字段命名。
2. 再补桌面工具日志文件与结果中心的结构化摘要。
3. 再补 Unity 运行时 API 的失败原因暴露。
4. 然后补 Unity 调试面板和 Inspector 展示。
5. 最后把自动化报告和对外交付材料也对齐到同一套字段口径。
