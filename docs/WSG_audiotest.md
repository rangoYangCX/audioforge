## 这套方案在 Unity 侧怎么理解

如果直接按 SDK 对接的视角理解，AudioForge 当前是一套典型的“工具端产出数据，Unity 运行时消费数据”的轻量音频中间件方案。

它不是把工具本身嵌进 Unity，也不是让程序直接读取工具工程，而是把两边明确拆开：

- 工具端负责维护内容、整理规则、做导出前检查，并产出稳定交付物。
- Unity 端负责消费导出结果、完成运行时播放和接入项目自身资源方案。
- 双方通过导出结果协作，而不是共享工具内部实现。

如果参考 FMOD 或 Wwise 这类中间件的理解方式，这套方案的基本思路是一致的，只是当前做的是轻量版本：

- 工具负责内容生产和规则收口。
- 导出物负责成为程序侧的稳定边界。
- 运行时负责执行，不反向依赖工具内部实现。

因此，从对接角度看，当前已经具备一条完整链路：

- 在工具端维护音频内容和工程结构
- 对内容做校验、试听和基础问题检查
- 将结果导出成 Unity 可消费的稳定交付物
- 提供一套参考运行时和验证材料，方便程序侧接入和联调

对程序同学来说，真正要抓住的是：接入时围绕导出结果工作，而不是围绕工具内部实现工作。

## 工具端和 Unity 端分别负责什么

如果按照职责拆分，可以这样理解。

### 工具端负责什么

- 维护事件、片段、基础结构和项目内音频内容。
- 在导出前完成整理、试听、静态校验和问题收口。
- 生成运行时可消费的交付结果。
- 为程序联调提供参考运行时材料和验证说明。

### Unity 端负责什么

- 加载导出结果，并建立项目里的运行时消费逻辑。
- 在游戏逻辑里触发播放、管理实例状态并接入项目自身资源加载方案。
- 在目标工程里完成最终联调和验收。

如果用当前 SDK 文档里的命名来帮助理解，可以把 Unity 侧大致看成几块：

- AudioForgeDatabase：负责读取导出主数据，并把事件和总线等运行时信息组织起来。
- AudioForgeRuntime：作为游戏代码调用音频能力的统一入口。
- AudioForgeVoiceManager：负责并发、冷却、顺序状态和活动实例这一类运行时控制问题。
- IAudioForgeResourceProvider：负责把项目里的资源加载方案接进来。
- AudioForgeBusState：负责总线状态这一层运行时控制。

### 明确的边界是什么

- Unity 侧不应该直接依赖工具工程本身。
- 运行时输入应以导出结果为准，而不是在项目里另维护一套脱节规则。
- 真正的接入细节放在主对接文档里，这份文档只帮助先建立正确理解。

## 可以对照其他中间件怎么理解

为了方便理解，可以用成熟中间件常见的几层能力来做对照。

### 1. 对照内容编辑端理解

如果参考 FMOD Studio 或 Wwise Authoring，可以把 AudioForge 当前看成一个轻量内容编辑端：

- 重点在于维护内容，而不是把规则散在工程代码里。
- 重点在于让音频同学有独立工作入口，而不是频繁依赖程序侧手改配置。

### 2. 对照构建和导出理解

如果参考 Build、Bank、SoundBank 这类概念，可以把它理解成一套稳定导出能力：

- 工具端负责把维护结果收敛成程序可消费的输入。
- 程序侧围绕统一交付物开发，而不是每次靠人工同步规则。

### 3. 对照运行时理解

如果参考 Runtime 层，可以把 Unity 侧理解成消费导出结果并完成运行时执行的一层：

- 负责播放。
- 负责状态管理。
- 负责资源接入。
- 负责在目标项目里完成最终行为验证。

如果想和 SDK 名称对应起来理解，可以简单记成：

- AudioForgeRuntime 更像对外入口。
- AudioForgeDatabase 更像运行时数据底座。
- AudioForgeVoiceManager 更像轻量的触发和实例控制层。
- IAudioForgeResourceProvider 更像资源接入抽象层。
- AudioForgeBusState 更像总线状态控制层。

### 4. 对照验证与联调理解

如果参考成熟中间件项目里的验证链路，AudioForge 当前也已经有对应的轻量形态：

- 工具端先做基础校验和试听。
- 程序侧再基于统一材料接入和联调。
- 交付前后可以围绕统一结果做复核和回查。

在当前仓库里，这一层也有对应的参考命名，方便组内识别：

- AudioForgeStreamingAssetsProvider：用于说明一类典型资源接入方式。
- AudioForgeRuntimeDebugPanel：用于联调时观察运行时状态和行为结果。
- AudioForgeEventPlayer / AudioForgeBootstrap：用于说明最小触发链路和验证入口怎么组织。

## 当前工作流怎么理解

如果把它放进实际项目流程里，可以按 SDK 对接视角理解成五步。

### 1. 内容先在工具端收口

音频同学先在工具里维护事件、片段、分类和基础结构，把原本容易散在 Excel、命名约定、程序配置里的内容收回到一个统一入口。

### 2. 在工具端完成整理和导出前检查

内容整理完后，可以在工具端继续做导入整理、批量处理、试听和基础问题检查。这样做的价值是把一部分问题前移，而不是等接进 Unity 后再集中返工。

### 3. 工具导出统一交付物

当内容收敛到可以交接的状态后，工具负责导出运行时需要消费的结果。对程序侧来说，这一步的意义在于接入输入被固定下来，后面不需要再反过来依赖工具内部工程结构。

### 4. Unity 侧围绕导出结果完成接入

程序同学在 Unity 里要做的，不是重建一套内容编辑体系，而是围绕导出结果完成运行时消费、播放触发、资源加载和项目内联调。

### 5. 双方用统一材料做联调和验收

当工具端、导出物和运行时都已经接起来后，最后一步就是围绕统一文档、验证材料和检查结果做联调。这一点和成熟中间件项目的思路是一样的，只是当前方案更轻、更聚焦当前项目范围。

## 从运行时流程看，event 和源音频是什么关系

如果从 SDK 实际消费流程来理解，可以把它拆成一条比较直观的链路：

1. 游戏代码先触发一个 event。
2. 运行时根据这个 event 找到它对应的播放规则和候选内容。
3. event 再去关联一个或多个源音频条目，而不是直接把某一条音频硬写死在业务代码里。
4. 运行时根据当前规则选出最终要播放的内容，并结合总线、实例状态和资源加载结果完成播放。

这样理解会更接近 FMOD/Wwise 里的使用习惯：

- event 更像播放入口和行为规则的承载体。
- 源音频更像被 event 组织和调度的实际内容。
- SDK 的职责不是替代内容编辑，而是把“触发什么、怎么选、怎么播”这条链路在 Unity 里跑通。

对程序侧来说，最关键的不是单独盯某一个音频文件，而是围绕 event 去理解运行时行为，因为真正进入游戏逻辑的是 event，源音频只是被它调度的底层资源。

## 当前大致支持哪些编辑能力

如果按组内最关心的功能层面来讲，当前方案已经支持一批休闲项目里最常用的编辑能力。

### 1. event 层编辑

- 可以给一个 event 组织多个候选源音频，而不是只能绑定单条声音。
- 可以维护 event 的基础归类和所属总线。
- 可以控制 event 的播放方式，当前重点覆盖的是随机、顺序和连击这几类常见触发模式。
- 可以维护避免连续重复、触发冷却、实例上限这类常见运行时约束。

如果用成熟中间件的习惯来理解，这一层就是把“一个声音触发点该怎么播”收口在 event 上，而不是分散到业务代码里。

### 2. 源音频层编辑

- 可以把多个源音频组织到同一个 event 下。
- 可以维护候选源音频之间的权重关系，用于随机挑选时控制命中倾向。
- 可以保留与裁剪、循环相关的内容信息，作为后续扩展和运行时的基础。

这层能力的价值在于，程序接入时面对的是已经整理好的内容集合，而不是一堆零散 wav 文件。

### 3. 常见播放变化编辑

- 支持基础音量和音高这一类常见播放调整。
- 支持随机音量、随机音高这类让重复音效不那么机械的常见变化。
- 支持连击场景下的递进式变化理解，可以覆盖一些轻量反馈型玩法。

这部分可以类比成熟中间件里最常用的基础 modulation 能力，只是当前版本聚焦在休闲项目高频够用的范围内。

### 4. 常见播放模式编辑

- Random：适合一组相近音效做离散随机播放。
- Sequence：适合按固定顺序轮播内容。
- Combo：适合连续触发时做递进反馈。

这三类模式基本覆盖了当前项目里最常见的一批交互音频需求，也是当前 SDK 对接时最值得优先理解的行为层。

### 5. 总线和运行时控制相关编辑

- 可以给内容归到基础总线结构里。
- 可以让运行时恢复总线初始状态，并在播放时叠加总线层控制。
- 可以把内容层编辑和总线层控制拆开，避免所有事情都堆在 event 本身。

这部分对应的是中间件里常见的 bus 管理思路，只是当前范围控制得更轻。

## 当前这套方案已经能支持什么

- 围绕 event 组织和消费音频内容。
- 让一个 event 对接多个候选源音频。
- 处理随机、顺序、连击这类常见播放逻辑。
- 处理随机变化、冷却、实例限制和总线控制这类常见运行时约束。
- 让 Unity 侧围绕统一导出结果完成播放、资源接入和联调。

如果结合当前 SDK 命名来理解，上面这些能力最终会落实到几类可识别对象上：

- 有运行时入口层，用来承接游戏代码调用。
- 有数据组织层，用来消费导出结果。
- 有实例控制层，用来处理触发后的运行时状态。
- 有资源接入层，用来适配项目自己的资源方案。
- 有调试观察层，用来支持联调和回查。

## 对协作的实际意义

### 1. 音频维护有统一入口

- 音频内容可以集中在工具端整理和管理。
- 这能减少内容维护分散、命名不统一、规则口头传递的问题。
- 对音频组来说，至少可以把大量重复性的整理和收口工作放在一个可复用入口里完成。

### 2. 程序接入有统一输入

- Unity 侧只需要围绕导出结果开发和联调。
- 这样更容易把接入工作做成一条清晰链路，而不是每次重新解释规则。
- 这也意味着程序侧不需要频繁介入内容细节本身，分工会更清楚。

### 3. 交付和联调更容易留痕

- 当前仓库除了工具和运行时材料，还保留了验证脚本、检查报告和主对接文档。
- 这意味着它已经可以支撑组内交付、回查和联调，而不是只有一个能运行的工具壳子。
- 这一点也是它和普通内部小工具的区别之一：不是只有“能用”，而是开始具备“可交付、可回查、可协作”的形态。
