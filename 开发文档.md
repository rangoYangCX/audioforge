1. 文档定位

本文档用于定义 AudioForge 的产品边界、数据契约、导出规范、Unity 对接约束、构建流程与验收标准。

AudioForge 是一个面向休闲游戏和中轻度项目的定制化游戏音频中间件工具。它的目标不是替代 Wwise/FMOD 的全栈能力，而是在不依赖 Unity 引擎编辑器扩展代码的前提下，提供一套可由音频设计人员独立维护、由运行时读取执行的数据驱动音频工作流。

本文档的重点不是 UI 效果，而是“可长期维护的工具与运行时契约”。所有实现都必须优先服从数据稳定性、导出可复现性与运行时行为一致性。

2. 产品目标与非目标

2.1 产品目标

- 提供类似 Wwise 的事件式音频配置体验，降低音频 TA 学习成本。
- 将音频逻辑从 Unity 场景脚本和业务代码中解耦，避免事件规则散落在 MonoBehaviour 中。
- 通过导出的静态数据与资源包驱动 Unity 运行时，实现工具侧和游戏侧的低耦合协作。
- 让音频设计修改参数后无需改动 Unity 运行时代码即可完成绝大多数日常调优。
- 保证导出数据具备稳定的版本控制、可审阅性和可回滚性。

2.2 非目标

- 不实现底层 DSP、混响图、实时混音台或复杂插件系统。
- 不实现 Unity Editor 深度嵌入，不依赖自定义 Inspector、AssetPostprocessor 或 Timeline 扩展。
- 不直接负责 Addressables、AssetBundle 或热更新系统的最终加载策略，但需为这些系统预留兼容字段。
- 不以“试听效果完全等同于 Unity 原生音频管线”为前提，试听只需在事件选择和参数计算层面保持一致。

3. 核心原则

3.1 数据优先

所有运行时行为必须由可导出的结构化数据驱动，不允许将事件选择逻辑、权重、冷却、连击规则硬编码在 Unity 业务层。

3.2 工具与运行时解耦

AudioForge 负责编辑、校验、导出；Unity 运行时负责加载、缓存、播放、状态管理。两端通过明确的数据契约通信，而不是通过共享编辑器脚本或内嵌 Python 逻辑耦合。

3.3 行为可定义

Random、Sequence、Combo、Max Instances、Cooldown 等行为必须在文档层定义精确语义，避免工具试听与运行时实现各自理解。

3.4 导出可复现

同一份工程在相同输入下多次导出时，除时间戳等显式动态字段外，产物内容必须一致。这是后续做版本对比、CI 校验和回归验证的基础。

3.5 分层明确

项目必须严格遵守 MVC。Model 不依赖具体 QWidget；View 不直接修改业务数据；Controller 负责连接交互与状态变更。

4. 总体架构

4.1 工具架构

- Model：维护工程树、事件数据、剪辑数据、校验结果、脏状态和导出快照。
- View：基于 PySide6 提供树、属性面板、内容列表、构建日志、校验面板等可视化界面。
- Controller：接收 UI 信号，执行业务命令，调用校验器、导出器和试听服务。
- Services：承载与 UI 无关的功能模块，如序列化、枚举生成、资源收集、试听模拟、日志写入、Schema 校验。

4.2 Unity 运行时架构

- AudioDatabase：读取 AudioData.json，并构建内存索引。
- AudioRuntime：对外暴露 PlayEvent、StopEvent、SetBusVolume 等接口。
- AudioVoiceManager：负责实例数限制、冷却判定、组合状态和正在播放的实例管理。
- AudioResourceProvider：按资源标识解析并加载 AudioClip，可由 Resources、Addressables、AssetBundle 或自定义加载器实现。
- AudioBusState：维护各 Bus 的音量、静音、淡入淡出状态。
- AudioGameSyncRegistry：phase3 新增，维护 Game Parameter、State Group、Switch Group 的静态定义与默认值。
- AudioEmitterContext：phase3 新增，作为 game object / emitter 维度的运行时上下文，承载 per-emitter RTPC 与 Switch 当前值。
- AudioGameSyncEvaluator：phase3 新增，在事件触发前按当前 Game Sync 上下文求值，决定属性覆盖、分支切换与最终候选 Clip 集合。

5. Unity 对接边界

5.1 工具职责

- 提供事件编辑界面。
- 保存工程文件。
- 对输入数据做静态校验。
- 导出扁平化运行时数据。
- 导出可选的资源清单与代码枚举。
- 提供基于相同规则的本地试听。

5.2 Unity 运行时职责

- 解析导出数据并完成索引初始化。
- 根据事件 ID 查找配置并执行实际播放。
- 管理实例并发限制、冷却时间与序列游标。
- 将逻辑音量和音高转换为 Unity 可执行参数。
- 根据项目需要选择同步预加载、懒加载或异步加载资源。

5.3 明确禁止项

- 禁止要求 Unity 侧读取 .afproj 工程文件。
- 禁止将音频逻辑配置写回 Unity 场景或 Prefab 作为唯一数据源。
- 禁止让 Python 工具生成依赖 Unity Editor API 的代码作为主流程前提。
- 禁止在业务代码中复制实现一套与导出数据脱节的随机或连击规则。

6. 项目结构建议

建议采用以下目录结构：

```text
AudioForge/
  app/
    controllers/
    models/
    services/
    views/
    widgets/
    schemas/
    utils/
  tests/
    unit/
```

模块间依赖方向必须保持为 views -> controller -> services/model，禁止 views 直接依赖 services 中的导出细节并自行修改持久化数据。

7. UI 设计规范

7.1 主窗口布局

主窗口采用嵌套 QSplitter，并在当前阶段进一步收敛为 Wwise 兼容工作台：左侧工程树、中部主编辑工作台、底部结果坞；工程摘要、当前对象与对象引用按工作区上下文栏提供，不再固定占用独立右侧栏。

其中：

- 左侧工程树负责对象定位与层级组织。
- 中部工作台负责当前任务的核心编辑动作。
- 工作区上下文栏负责显示任务导航，以及按需显示工程摘要、当前对象与对象引用。
- 底部结果坞负责日志、校验、构建和响度结果的快速回看。

工程页中的 Bus 编辑需采用类似 Wwise 的分栏方式：左侧为可拖拽重排的 Bus 层级，右侧为主 Bus 固定区、属性编辑器与传输控制监视区。Bus 不再只是字符串列表，而是可持久化的工程对象，至少包含 Bus 名称、父 Bus、基础音量和静音状态。

二期中，左侧入口应升级为固定三分页的“对象浏览器”：第一页为总线树，第二页为源音频树，第三页为事件树。总线树负责总线创建、父子层级调整、输出路由浏览和有效输出计算；源音频树负责按编辑期路径浏览与定位源文件、显示引用状态和缺失状态；事件树负责 Folder / Event 浏览、事件类型识别与 Source Binding 悬浮窗入口，不与总线节点或源音频节点混排。

Bus 树中的有效输出计算按以下规则展示：

- Effective Output Target：当前 Bus 若显式指定输出目标，则取该目标；否则沿父 Bus 链向上继承，直到 Master。
- Effective Volume Db：当前 Bus 基础音量与所有祖先 Bus 基础音量累加，仅用于编辑器试听和诊断显示。
- Effective Mute：当前 Bus 或任一祖先 Bus 为静音时，结果视为静音。
- Effective Path：UI 中需明确显示“当前 Bus -> 父 Bus -> ... -> Master”的完整链路，避免只给最终结果不显示推导来源。

当前实现已额外具备以下交互细节：

- 对象头部会持续显示当前对象、Bus 视图状态、当前页签、报告页签、脏状态与片段选择数。
- 属性页、内容页、报告页与关键 splitter 的导航状态会在对象刷新后恢复，避免切页后布局漂移。
- 主窗口、分离工程浏览器、设置窗以及各工作区命名 splitter 的几何和尺寸都会写入偏好，重开后恢复到上次使用状态。
- 工程树与片段表均提供右键菜单与高频快捷键，适合高频编辑场景。
- 问题中心采用“列表 + 详情”结构，支持跳转对象、保持选中项与滚动位置。

7.2 左侧工程浏览器

组件：QTreeView + 自定义 Model，优先于 QStandardItemModel，以保证后续扩展拖拽、排序、虚拟节点与批量刷新能力。二期中，该区域应以“对象浏览器”容器承载总线树 / 源音频树 / 事件树三分页，三页共用筛选框与顶部上下文栏，但数据模型隔离；创建入口和右键菜单骨架可复用，动作集按分页裁剪。

节点类型：

- Folder：仅用于编辑期组织，不进入运行时数据。
- Event：可导出运行时事件。
- Source Binding：挂在 Event 下的源音频引用节点，代表“该事件引用了哪个源音频以及如何参与播放”，而不是新的 Event。
- Source Folder：仅存在于源音频树，用于按磁盘相对路径或逻辑分组浏览源文件，不进入运行时数据。
- Source Asset：仅存在于源音频树，对应 `asset_registry` 中的一个源路径；可显示引用次数、文件缺失与未引用状态，但不直接导出为新的运行时对象。

层级约束：

- Folder 只能包含 Folder 或 Event。
- Event 只能包含 Source Binding，不能包含 Event，也不能成为另一个 Event 的子节点。
- Source Binding 为叶子节点，不允许再挂载 Folder、Event 或其他 Source Binding。
- 总线树中的 Master / Bus 节点、源音频树中的 Source Folder / Source Asset 节点，以及事件树中的 Folder / Event / Source Binding 节点完全隔离，不允许互相拖拽穿透。

二期交互约束：

- 总线树、源音频树、事件树共享同一个浏览器容器，但分页切换不得改变当前中央工作区与已打开的编辑上下文；对象浏览器只是定位入口，不应强制打断主编辑流。
- 源音频树中的 Source Asset 拖拽到 Event 上时，应创建新的 Source Binding，而不是复制物理文件，也不是把源音频提升为 Event。
- 双击或回车聚焦 Source Asset 时，应优先切到资源工作区或定位到当前事件上下文中的对应片段；若当前没有引用关系，则至少给出文件定位或引用为空提示。
- 源音频树需要显示最少三类编辑期状态：文件缺失、未被任何事件引用、被多个事件复用。这三类状态默认只属于工具诊断，不进入当前导出结果。
- 事件设计页需要保留一个“源音频绑定”摘要区，但该区域默认只读，只负责显示当前事件下有哪些绑定音频、各自的 Enabled / Active 状态和来源路径，不再承担主编辑入口。
- 当前版本里，真正的绑定编辑入口统一收敛到事件树展开后的 Source Binding 悬浮窗；追加绑定、拖拽绑定、Enabled / Active 切换都在该悬浮窗中完成，事件设计页摘要区只做结果回显。
- 将源音频拖拽到 Event 上时，应创建新的 Source Binding，而不是复制物理文件，也不是把源音频提升为 Event。
- 删除 Source Binding 只解除事件与源音频的关联，不删除源文件，也不影响其他 Event 对同一源音频的引用。
- 当前实现中，事件树默认只显示 Folder / Event，并按 OneShot、Random、Sequence、Combo 四类事件显示差异化图标；展开事件节点时会弹出 Source Binding 悬浮窗，而不是继续把 binding 子节点塞进树里。
- 当前实现中，从源音频树拖到 Event 时会即时创建对应绑定；事件树绑定悬浮窗中的拖放会明确反馈“追加成功/跳过重复项”，底层数据仍然只回写 `Event.clips`。
- 当前实现中，Source Binding 的 Active / Enabled 已作为 editor-only 状态挂在 clip 上，由悬浮窗承载切换；试听、导出与校验统一按“有效绑定集合”消费，不直接把这些字段写进当前 Unity 运行时契约。
- Source Binding 需要同时具备 `Enabled` 与 `Active` 两类状态：`Enabled` 表示该绑定是否允许参与当前事件；`Active` 表示该绑定当前是否属于有效播放集合。Disabled 绑定会自动视为非 Active。
- OneShot 事件必须且只能有一个 `Active` Source Binding；Random / Sequence / Combo 允许同时存在多个 `Active` 绑定，并且新追加或重新启用的绑定默认会进入 Active，便于直接参与试听、导出与校验。
- 事件列表行首需按事件类型显示图标和次级状态徽标，保证用户在树上不展开属性面板也能快速识别事件类型。

能力要求：

- 新建、重命名、删除、复制、粘贴。
- 支持拖拽调整层级。
- 支持搜索过滤。
- 支持多选批量删除和批量改名预处理。
- 支持脏状态提示和非法命名即时反馈。
- 源音频树至少支持按目录折叠、关键字过滤、状态过滤、定位磁盘文件、查看引用事件数量与缺失状态，以及从源音频定位到引用事件或追加到当前事件。
- 当前实现已补充：右键菜单、Enter 聚焦编辑器、F2 重命名、Delete 删除、Ctrl+C 复制对象标识、事件树多选、批量改总线、批量重命名、批量删除，以及基于当前过滤词的事件搜索定位。

8. 领域模型设计

8.1 核心实体

Folder

- id：编辑期唯一标识。
- name：显示名称。
- children：Folder 或 Event 节点。

Event

- id：运行时唯一标识，需全局唯一。
- display_name：编辑器显示名，可为空。
- bus：所属总线。
- play_mode：Random、Sequence、Combo。
- avoid_immediate_repeat：是否避免连续命中同一 Clip。
- volume_db：基础音量。
- volume_rand_min_db、volume_rand_max_db：音量随机范围。
- pitch_cents：基础音高。
- pitch_rand_min_cents、pitch_rand_max_cents：音高随机范围。
- max_instances：并发实例数限制。
- cooldown_seconds：事件冷却时间。
- steal_policy：ReachedLimit 时的处理策略。
- combo_pitch_step_cents：Combo 每步附加音高。
- combo_reset_seconds：Combo 重置时间。
- combo_max_step：Combo 最大累计步数。
- load_policy：资源加载策略。
- clips：Clip 列表。
- notes：备注。

Clip

- id：事件内唯一标识，建议全局也唯一。
- source_path：源文件绝对路径或工程相对路径。
- export_path：导出资源目标相对路径。
- asset_key：运行时资源键。
- weight：权重。
- trim_start_ms：裁剪起点。
- trim_end_ms：裁剪终点。
- tags：标签列表。

二期扩展实体（编辑期）：

SourceAsset

- id：源音频资源唯一标识。
- source_path：物理源文件路径。
- display_name：用于源列表和拖拽提示的显示名。
- analysis_summary：源文件时长、声道、采样率、响度摘要等缓存信息。

EventSourceBinding

- id：事件内绑定唯一标识。
- event_id：所属 Event。
- source_asset_id：指向 SourceAsset 的引用。
- enabled：是否参与当前事件的候选集合。
- active：是否属于当前有效播放集合。OneShot 下只能有一个为真；Random / Sequence / Combo 下可同时有多个为真。
- order：在 Event 内的显示和 Sequence 顺序。
- weight：Random / Combo 模式下的权重。
- trim_start_ms、trim_end_ms、fade_in_ms、fade_out_ms：绑定级修剪与包络设置。
- notes：绑定级备注。

兼容策略：一期的 Clip 可视为 EventSourceBinding 的 V1 形态；二期若引入 SourceAsset / EventSourceBinding 拆分，必须保持 ClipId、AssetKey 和既有导出键稳定，避免影响 Unity 对接。

8.2 命名规范

- Event ID 必须全局唯一。
- Event ID 只允许字母、数字和下划线，且必须以字母开头。
- 推荐格式：系统_功能_语义，例如 UI_Click_Normal、BGM_Lobby_Main、SFX_Hit_Light。
- Display Name 可使用中文，仅用于编辑器显示。
- Bus 名称必须来自受控枚举，不允许自由文本。

9. 行为语义定义

9.0 二期事件类型规划（编辑器设计冻结）

- 二期事件类型暂定为 OneShot、Random、Sequence、Combo。
- 创建 Event 时必须允许直接选择类型，并在树、右键菜单和新建向导中保持同一套命名。
- 当前 9.1 至 9.6 章节仍是一期已实现行为基线；二期未落地的类型和字段，在 Unity 运行时准备完毕前不得写入当前导出契约。

OneShot：

- 表示一次触发一次播放的基础事件类型。
- 允许勾选 Loop；勾选后按循环事件处理，直到收到 Stop 或切换逻辑为止。
- 可挂载多个 Source Binding，但运行时候选必须只取当前 `Active` 的那一个，其他绑定用于 A/B 切换、版本保留或人工候补。
- 当前实现已落地 OneShot 的编辑器类型、树图标和本地试听/导出/校验语义；运行时仍只接收过滤后的有效 clip 集合，不接收 editor-only 状态字段本身。

Random：

- 表示从多个处于 `Active` 且 `Enabled` 的 Source Binding 中按权重随机选择。
- 当前已实现的随机语义继续保留；二期主要补齐图标、类型创建入口和绑定层级表现。

Sequence：

- 表示按 Event 内绑定顺序依次播放。
- 除现有顺序模式外，二期需补充 `ShuffleBag` 选项：每轮从所有 `Active` 且 `Enabled` 的绑定中无重复抽取一次，耗尽后再重建下一轮。
- Sequence 的顺序基于 EventSourceBinding.order，而不是源文件导入时间。

Combo：

- 基于现有 Combo 语义继续增强，保留事件级 step 状态机。
- 二期先复用当前 Combo 参数，不额外拆出新的导出类型；绑定选择默认按 `Active` 且 `Enabled` 的集合参与，后续如需 Combo + Sequence 子模式，应单独立项。

校验规则补充：

- OneShot 至少有 1 个 `Enabled` 绑定，且必须恰好有 1 个 `Active` 绑定。
- Random、Sequence、Combo 至少有 1 个同时处于 `Enabled` 和 `Active` 的绑定。
- Sequence 在 `ShuffleBag` 模式下禁止只有 0 个可用绑定；只有 1 个绑定时允许但应给出“洗牌无实际收益”的提示级警告。
- 事件类型变更时，不兼容参数应进入“保留但隐藏”或“引导迁移”状态，不能直接静默丢弃。

9.1 Random

- 在当前 Event 的 Clips 集合中按权重进行离散随机选择。
- Weight 的含义是未归一化权重，而非百分比。
- 若 avoid_immediate_repeat 为 true 且 Clip 数量大于 1，则本次候选集合需排除上次命中的 Clip。
- 若排除后集合为空，则回退为原始全集合。

9.2 Sequence

- Clips 按显示顺序形成序列。
- 每次触发时取当前游标对应 Clip，播放完成后或触发成功后将游标前进 1。推荐以“触发成功后前进”为标准。
- 游标达到尾部后回到 0。
- Sequence 游标默认按 Event 全局维护，而非按调用者实例维护。

9.3 Combo

- Combo 基于 Event 级状态机，不基于 Clip 状态机。
- 连续两次成功触发间隔小于等于 combo_reset_seconds 时，combo step 加 1；否则重置为 0。
- 实际播放音高 = 基础 Pitch + 随机 Pitch + step * combo_pitch_step_cents。
- 当 combo_max_step 大于 0 时，累计 step 不得超过该上限。
- Combo 模式下 Clip 选择默认仍按 Random 语义执行，除非后续增加 ComboSequence 子模式。

9.4 Max Instances

- 统计对象为同一 Event 正在播放中的实例数。
- 0 表示无限制。
- 当达到上限时，由 steal_policy 决定处理方式。

9.5 Steal Policy

- RejectNew：拒绝新播放请求。
- StopOldest：终止最早开始的实例并播放新实例。
- StopQuietest：预留策略，第一期不开放界面编辑，当前试听和 Unity 验证运行时均未实现。

9.6 Cooldown

- 以 Event 为粒度记录最后一次成功触发时间。
- 若本次触发时间与最后成功触发时间间隔小于 cooldown_seconds，则直接拒绝，不进入选 Clip 阶段。

11.3 字段契约

顶层字段：

- SchemaVersion：整数，必填。运行时据此决定兼容策略。
- GeneratedAt：ISO 8601 时间戳，必填。
- ProjectName：字符串，必填。
- Buses：字符串数组，必填。
- BusConfigs：总线对象数组，可选；若存在则应作为总线初始化状态的权威来源。
- Events：字典，键为 Event ID，必填。

BusConfig 字段：

- Name：字符串，必填。
- ParentBus：字符串，可选，默认 Master。
- VolumeDb：浮点数，可选，默认 0.0。
- IsMuted：布尔，可选，默认 false。

Event 字段：

- Bus：字符串，必填，必须存在于总线列表中。
- PlayMode：字符串，必填，可选值为 Random、Sequence、Combo。
- AvoidImmediateRepeat：布尔，可选，默认 false。
- VolumeDb：浮点数，可选，默认 0.0。
- VolumeRandDb：长度为 2 的数组，可选，默认 [0.0, 0.0]。
- PitchCents：整数，可选，默认 0。
- PitchRandCents：长度为 2 的数组，可选，默认 [0, 0]。
- MaxInstances：整数，可选，默认 0。
- CooldownSeconds：浮点数，可选，默认 0.0。
- StealPolicy：字符串，可选，默认 RejectNew。
- LoadPolicy：字符串，可选，默认 OnDemand；一期工具与 Unity 验证运行时固定按 OnDemand 工作。
- ComboPitchStepCents：整数，仅 Combo 模式使用。
- ComboResetSeconds：浮点数，仅 Combo 模式使用。
- ComboMaxStep：整数，可选，默认 0，表示无限累计。
- Clips：数组，必填，长度至少为 1。

Clip 字段：

- ClipId：字符串，必填。
- AssetKey：字符串，必填。Unity 运行时据此加载资源。
- Weight：整数，必填，范围 1 到 100，表示相对权重而非百分比。

11.4 版本兼容规则

- 运行时必须显式检查 SchemaVersion。
- 若版本高于当前支持上限，运行时应报错并拒绝继续初始化。
- 若版本低于当前版本但仍在兼容范围内，可走兼容适配器。
- 任何破坏性变更都必须升级 SchemaVersion。

11.5 二期编辑器演进兼容策略

- 左侧对象浏览器改为总线树 / 源音频树 / 事件树三分页，属于纯编辑器信息架构调整，不改变当前导出的 `Buses`、`BusConfigs`、`Events` 基本语义。
- 源音频树基于工程内 `asset_registry`、事件引用关系和文件存在性计算得出；这些索引、引用计数与缺失状态默认不新增到当前 `AudioData.json`、`AudioManifest.json` 顶层字段。
- 工程设置现已补充“根据事件命名智能分配总线”开关；它只影响工具端新建/导入事件时默认选中的 Bus，不新增 Unity 运行时字段。
- 事件设计页保留“源音频绑定”只读摘要区，真正的编辑入口统一收敛到事件树展开后的 binding 弹窗；追加、替换和 Active / Enabled 调整都只在工具层重组 `Event.clips`，不改变 Unity 侧资源键语义或输入协议。
- 事件树中的 Source Binding 可视化、从源音频树拖到 Event 创建绑定、以及展开事件节点后弹出的 binding 悬浮窗，本质上都只操作 `Event.clips` 与 `asset_registry`，不新增 Unity 侧运行时字段或导出协议。
- 从源音频拖拽到事件树自动建 Event 时，若生成后的事件只包含 1 个源音频，当前默认落为 `OneShot`；这仍然只通过既有 `PlayMode + Clips` 契约表达，不引入新的运行时结构。
- `Effective Output Target`、`Effective Volume Db`、`Effective Mute` 和完整路由链属于编辑器诊断结果，默认不新增到当前 `AudioData.json` 顶层字段。
- `SourceAsset`、`EventSourceBinding`、`Active`、`Enabled`、`EventType`、`ShuffleBag` 等二期字段，在 Unity 运行时与 SchemaVersion 升级方案确认前，只允许存在于 `.afproj` 或 feature flag 中；当前导出仅允许通过“有效绑定过滤结果”间接体现，不得把 editor-only 状态字段原样写入导出结果。
- 若后续确定要把 OneShot、Sequence 洗牌模式或绑定状态写入运行时契约，必须同步升级 SchemaVersion，并提供 Unity 端显式兼容实现与验证用例。

11.6 三期 Game Sync 契约现状

- 当前 `SchemaVersion` 已升级到 2，并新增项目级 `GameParameters`、`StateGroups`、`SwitchGroups` 顶层字段。
- 事件对象当前已新增 `RtpcBindings`、`StateOverrides`、`SwitchVariants` 和 `DefaultClipIds`；总线对象当前已新增 `RtpcBindings` 与 `StateOverrides`，同时保持 `BusConfigs`、`Events`、`Clips` 既有语义不变。
- `StateGroup` 与 `SwitchGroup` 当前同时保留 names list 与 `state_effects` / `switch_effects` 子项效果映射，兼容旧工程的同时，让 authoring、preview、exporter 与 Unity runtime 都能消费每个子项的 volume / pitch / mute / notes。
- `SetState` 当前按全局语义生效；`SetSwitch` 与 emitter / game object 绑定；RTPC 当前同时支持全局值与 emitter 局部值。这三类控制量在运行时存储上保持分离。
- 当前 Runtime/Preview 已实现触发瞬间的 Game Sync 求值；若后续要接近 Wwise 的持续实时调制，则仍需扩展活动声部结构和运行时重算链路。
- 详细字段、当前 API 与求值顺序见 `docs/UnityRuntime三期GameSync设计.md`。

11.6 二期对象浏览器对标结论（不改 SDK）

- 借鉴 Wwise：延续对象浏览器常驻左侧、总线与事件分离、路由链可解释的工作台心智，但不复制其完整对象宇宙，也不提前引入 SoundBank 语义。
- 借鉴 FMOD Studio：把筛选、分页切换、对象定位和上下文连续性做顺，而不是把所有信息都堆到属性面板；源音频树和事件树分离后，中央编辑区仍保持任务聚焦。
- 借鉴 CRI ADX2：用独立源音频树承载素材索引、缺失检测、复用关系和大批量浏览能力，但索引仍留在工具工程里，不要求 Unity 运行时识别这些资产管理结构。
- 当前不借鉴的部分：运行时 profiler 协议、实时远程调试桥、SoundBank / Bank 构建策略、DSP 图编辑和插件链配置。这些都会改变当前 Unity 对接复杂度，不属于本轮范围。
- 本轮最重要的边界是：三分页对象浏览器只改变编辑器的信息架构和浏览效率，不改变 Unity 的输入语义、资源加载键和现有 SDK 对接代码。

12. 资源导出规范

12.1 资源身份

资源必须同时具备以下三个层次的身份：

- ClipId：逻辑片段标识。
- AssetKey：运行时加载键。
- SourcePath：编辑期源文件路径。

SourcePath 可以变，AssetKey 应保持稳定；AssetKey 可以映射到不同物理打包系统；ClipId 则用于事件内部管理和调试。

12.2 导出目录建议

```text
Export/
  AudioData.json
  AudioEventID.cs
  AudioManifest.json
  Assets/
    ui/
    battle/
    bgm/
```

12.3 AudioManifest.json

建议增加资源清单，用于 Unity 或 CI 校验导出资源一致性。字段建议包括：

- AssetKey
- SourcePath
- ExportPath
- FileSize
- Hash
- BuildFingerprint
- ReferencedByEvents

自 AudioForge 0.05 起，`BuildFingerprint` 用于描述资源内容级构建指纹，默认至少覆盖源文件 Hash、运行时格式、Trim、Fade 和 Loop 边界。它的用途是支持增量构建时安全复用旧导出资源，而不是改变 Unity 运行时消费方式。

12.4 资源复制规则

- 仅复制被 Event 引用到的资源。
- 相同 AssetKey 对应不同源文件时，视为 Error。
- 相同源文件被多个 Event 引用时，只导出一份物理文件。
- 默认覆盖同名目标文件，但需优先比对 `BuildFingerprint` 与导出状态，避免无意义重建。
- 自 AudioForge 0.05 起，增量构建允许从上一次成功导出结果中复用未变化资源，但整个导出仍必须先写入临时目录，再以原子替换方式提交，禁止保留半成品覆盖旧导出。

13. C# 代码生成规范

13.1 生成目标

为 Unity 端生成强类型事件入口，降低魔法字符串使用。

13.2 建议产物

- AudioEventID.cs：事件枚举。
- AudioEventNames.cs：枚举到字符串的映射辅助类，可选。

13.3 枚举生成规则

- 枚举成员来自全部 Event ID。
- 枚举生成顺序必须稳定，建议按 Event ID 字典序排序。
- 非法 C# 标识符需在校验阶段直接报错，不建议静默替换。

示例：

```csharp
public enum AudioEventID
{
    BTL_Combo_Hit,
    UI_Click_Normal
}
```

14. Build 管线设计

14.1 Build 阶段

Build 必须按以下顺序执行：

1. 保存内存数据快照。
2. 执行静态校验。
3. 解析构建范围（全量 / 增量 / 选中）。
4. 基于当前工程与上次成功导出结果生成 BuildPlan。
5. 生成运行时数据模型。
6. 生成 AudioData.json。
7. 生成代码文件。
8. 生成资源清单。
9. 按 BuildPlan 重建或复用音频资源。
10. 输出 Build 报告。

14.2 失败策略

- 若存在 Error 级问题，Build 必须终止，不允许导出不完整产物。
- 若仅存在 Warning，允许继续导出，但 Build 报告需明确列出。
- 任一步骤失败时，禁止保留部分新产物覆盖旧产物。建议写入临时目录并在成功后原子替换。

14.3 Build 报告

建议额外输出 BuildReport.json，包含：

- BuildId
- StartedAt
- FinishedAt
- ProjectVersion
- SchemaVersion
- EventCount
- ClipCount
- WarningCount
- ErrorCount
- ExportedFiles
- BuildPlan.RequestedScope
- BuildPlan.EffectiveScope
- BuildPlan.SelectionLabel
- BuildPlan.RebuiltAssetKeys
- BuildPlan.ReusedAssetKeys
- BuildPlan.RemovedAssetKeys
- BuildPlan.OutOfScopeDirtyAssetKeys

15. 数据校验规范

15.1 级别定义

- Error：阻止保存或阻止 Build。
- Warning：允许保存与 Build，但需要用户确认风险。
- Info：仅提示，不阻断流程。

15.2 必须实现的 Error 规则

- Event ID 为空。
- Event ID 重复。
- Event ID 含非法字符。
- Event 无 Clip。
- Clip 源文件不存在。
- Weight 小于 1。
- Bus 不在受控枚举内。
- VolumeRand 最小值大于最大值。
- PitchRand 最小值大于最大值。
- Combo 模式缺失 ComboPitchStepCents 或 ComboResetSeconds。
- 同一 AssetKey 指向多个不同源文件。
- 导出目录不可写。

15.3 必须实现的 Warning 规则

- Event 只有一个 Clip 但设置了 AvoidImmediateRepeat。
- Sequence 模式只有一个 Clip。
- Cooldown 大于 5 秒。
- MaxInstances 为 1 且 StealPolicy 非 RejectNew。
- 资源格式不是推荐格式。
- 导出资源路径存在大小写冲突风险。

15.4 即时校验与 Build 校验

- 即时校验：用户编辑字段时尽快反馈本地规则。
- Build 校验：在导出前重新做全量校验，保证最终产物可信。

16. 本地试听规范

16.1 目标

本地试听用于尽量还原 Unity 运行时的事件选择、参数计算与实例生命周期，作为策划与音频设计阶段的听感校对工具。

16.2 一致性要求

试听必须与运行时在以下逻辑上完全一致：

- Random 权重选择
- AvoidImmediateRepeat 规则
- Sequence 游标推进
- Combo step 累加与重置
- Cooldown 拒绝逻辑
- Max Instances 判定策略
- StopOldest 抢占策略
- 基于 Trim 的真实试听时长
- Master 与事件输出总线的预览音量、静音叠加
- 属性编辑器中所有音高相关参数在本地试听里默认按保时长变调处理
- Combo 连击步进按半音档位累加

16.3 可接受差异

以下差异允许存在：

- 音频解码库不同导致的波形细节差异
- Unity 音频导入设置导致的压缩或延迟差异
- 不同平台硬件缓冲带来的播放起始偏差
- 当前工具端仍未复刻 Unity 原生总线 DSP、效果器链与空间化处理
- 当前工具端已将基础音高、随机音高和 Combo 附加音高统一为保时长试听；Unity 运行时若仍直接使用 `AudioSource.pitch`，则会与工具端在片段时长变化上存在已知差异。

- 当前工具端已支持 Combo 连击加成的保时长变调试听；Unity 运行时若要与该听感完全一致，需后续接入同等的变调 DSP 或自定义保时长变调实现。

16.3.1 总线预览状态说明

- 工具端的“传输控制”仅用于本地试听与响度监视，不写入 .afproj 导出配置，也不参与 AudioData.json 生成。
- 传输控制包含一个隐式主 Bus 和工程中定义的输出 Bus；最终有效输出按主 Bus 与输出 Bus 线性乘算后，再叠加到事件音量上。

16.4 试听技术选型

- 第一优先：pygame.mixer，适合快速播放与原型阶段。
- 备选：pydub + simpleaudio，适合更灵活的片段处理。
- 不建议将试听实现和导出实现混写在同一 Controller 中，应提取独立 PreviewService。

17. Unity 运行时接口建议

运行时最小接口建议如下：

```csharp
public interface IAudioRuntime
{
    bool PlayEvent(string eventId);
    bool PlayEvent(AudioEventID eventId);
    void StopEvent(string eventId);
    void StopBus(string busName);
    void SetBusVolume(string busName, float volumeDb);
    bool HasEvent(string eventId);
    void Preload(string eventId);
    void Unload(string eventId);
}
```

实现约束：

- PlayEvent 的返回值表示是否成功接受播放请求，而不是是否已经播放完成。
- HasEvent 仅检查数据库中是否存在，不应触发加载。
- Preload 与 Unload 是资源层能力，不应改写事件数据本身。

18. 性能目标

以下目标适用于桌面开发环境第一期实现：

- 1000 个 Event、5000 个 Clip 的工程在 3 秒内完成打开。
- 全量校验在 2 秒内完成。
- 生成 AudioData.json 与代码文件在 2 秒内完成，不含资源复制。
- 左侧树展开和筛选在常规编辑操作下无明显卡顿。

若第一期无法完全满足，应在实现中至少保留异步校验、分段刷新和延迟加载的扩展点。

19. 可用性要求

- 支持 Undo/Redo。
- 支持自动保存或崩溃恢复。
- 支持脏状态提示。
- 支持最近工程列表。
- 支持搜索与过滤。
- 支持批量修改 Clip 权重。
- 支持复制 Event 并自动建议新 ID。

25. 结论

AudioForge 的核心价值不在于“做出一个像 Wwise 的界面”，而在于建立一条稳定的数据驱动音频生产链路。只要工具、导出数据和 Unity 运行时三者之间的职责边界、字段契约和行为语义足够清晰，这套方案就能长期维护，并且不会反向依赖引擎侧实现细节。
