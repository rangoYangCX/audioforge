1. 文档定位

本文档用于定义 AudioForge 的产品边界、数据契约、导出规范、Unity 对接约束、构建流程与验收标准。

AudioForge 是一个面向休闲游戏和中轻度项目的定制化游戏音频中间件工具。它的目标不是替代 Wwise/FMOD 的全栈能力，而是在不依赖 Unity 引擎编辑器扩展代码的前提下，提供一套可由音频设计人员独立维护、由运行时读取执行的数据驱动音频工作流。

本文档的重点不是 UI 效果，而是“可长期维护的工具与运行时契约”。所有实现都必须优先服从数据稳定性、导出可复现性与运行时行为一致性。

2. 产品目标与非目标

2.1 产品目标

- 提供类似 Wwise 的事件式音频配置体验，降低音频 TA 学习成本。
- 将音频逻辑从 Unity 场景脚本和业务代码中解耦，避免事件规则散落在 MonoBehaviour 中。
- 通过导出的静态数据与资源包驱动 Unity 运行时，实现工具侧和游戏侧的低耦合协作。
- 让音频设计修改参数后无需改动 Unity 运行时代码即可完成绝大多数日常调优。
- 保证导出数据具备稳定的版本控制、可审阅性和可回滚性。

2.2 非目标

- 不实现底层 DSP、混响图、实时混音台或复杂插件系统。
- 不实现 Unity Editor 深度嵌入，不依赖自定义 Inspector、AssetPostprocessor 或 Timeline 扩展。
- 不直接负责 Addressables、AssetBundle 或热更新系统的最终加载策略，但需为这些系统预留兼容字段。
- 不以“试听效果完全等同于 Unity 原生音频管线”为前提，试听只需在事件选择和参数计算层面保持一致。

3. 核心原则

3.1 数据优先

所有运行时行为必须由可导出的结构化数据驱动，不允许将事件选择逻辑、权重、冷却、连击规则硬编码在 Unity 业务层。

3.2 工具与运行时解耦

AudioForge 负责编辑、校验、导出；Unity 运行时负责加载、缓存、播放、状态管理。两端通过明确的数据契约通信，而不是通过共享编辑器脚本或内嵌 Python 逻辑耦合。

3.3 行为可定义

Random、Sequence、Combo、Max Instances、Cooldown 等行为必须在文档层定义精确语义，避免工具试听与运行时实现各自理解。

3.4 导出可复现

同一份工程在相同输入下多次导出时，除时间戳等显式动态字段外，产物内容必须一致。这是后续做版本对比、CI 校验和回归验证的基础。

3.5 分层明确

项目必须严格遵守 MVC。Model 不依赖具体 QWidget；View 不直接修改业务数据；Controller 负责连接交互与状态变更。

4. 总体架构

4.1 工具架构

- Model：维护工程树、事件数据、剪辑数据、校验结果、脏状态和导出快照。
- View：基于 PySide6 提供树、属性面板、内容列表、构建日志、校验面板等可视化界面。
- Controller：接收 UI 信号，执行业务命令，调用校验器、导出器和试听服务。
- Services：承载与 UI 无关的功能模块，如序列化、枚举生成、资源收集、试听模拟、日志写入、Schema 校验。

4.2 Unity 运行时架构

- AudioDatabase：读取 AudioData.json，并构建内存索引。
- AudioRuntime：对外暴露 PlayEvent、StopEvent、SetBusVolume 等接口。
- AudioVoiceManager：负责实例数限制、冷却判定、组合状态和正在播放的实例管理。
- AudioResourceProvider：按资源标识解析并加载 AudioClip，可由 Resources、Addressables、AssetBundle 或自定义加载器实现。
- AudioBusState：维护各 Bus 的音量、静音、淡入淡出状态。

5. Unity 对接边界

5.1 工具职责

- 提供事件编辑界面。
- 保存工程文件。
- 对输入数据做静态校验。
- 导出扁平化运行时数据。
- 导出可选的资源清单与代码枚举。
- 提供基于相同规则的本地试听。

5.2 Unity 运行时职责

- 解析导出数据并完成索引初始化。
- 根据事件 ID 查找配置并执行实际播放。
- 管理实例并发限制、冷却时间与序列游标。
- 将逻辑音量和音高转换为 Unity 可执行参数。
- 根据项目需要选择同步预加载、懒加载或异步加载资源。

5.3 明确禁止项

- 禁止要求 Unity 侧读取 .afproj 工程文件。
- 禁止将音频逻辑配置写回 Unity 场景或 Prefab 作为唯一数据源。
- 禁止让 Python 工具生成依赖 Unity Editor API 的代码作为主流程前提。
- 禁止在业务代码中复制实现一套与导出数据脱节的随机或连击规则。

6. 项目结构建议

建议采用以下目录结构：

```text
AudioForge/
  app/
    controllers/
    models/
    services/
    views/
    widgets/
    schemas/
    utils/
  tests/
    unit/
    integration/
    fixtures/
  tools/
  exports/

```

模块间依赖方向必须保持为 views -> controller -> services/model，禁止 views 直接依赖 services 中的导出细节并自行修改持久化数据。

7. UI 设计规范

7.1 主窗口布局

主窗口采用嵌套 QSplitter，实现近似 Wwise 的三段式布局：左侧工程树、右上属性编辑、右下内容列表。底部可选日志面板与校验面板。

工程页中的总线编辑需采用类似 Wwise 的分栏方式：左侧为可拖拽重排的总线树，右侧为 Master 固定区、总线属性编辑器与试听监视区。总线不再只是字符串列表，而是可持久化的工程对象，至少包含名称、父总线、基础音量和静音状态。

当前实现已额外具备以下交互细节：

- 对象头部会持续显示当前对象、总线浏览状态、当前页签、报告页签、脏状态与片段选择数。
- 属性页、内容页、报告页与关键 splitter 的导航状态会在对象刷新后恢复，避免切页后布局漂移。
- 工程树与片段表均提供右键菜单与高频快捷键，适合高频编辑场景。
- 问题中心采用“列表 + 详情”结构，支持跳转对象、保持选中项与滚动位置。

7.2 左侧 Project Explorer

组件：QTreeView + 自定义 Model，优先于 QStandardItemModel，以保证后续扩展拖拽、排序、虚拟节点与批量刷新能力。

节点类型：

- Folder：仅用于编辑期组织，不进入运行时数据。
- Event：可导出运行时事件。

能力要求：

- 新建、重命名、删除、复制、粘贴。
- 支持拖拽调整层级。
- 支持搜索过滤。
- 支持多选批量删除和批量改名预处理。
- 支持脏状态提示和非法命名即时反馈。
- 当前实现已补充：右键菜单、Enter 聚焦编辑器、F2 重命名、Delete 删除、Ctrl+C 复制对象标识、事件树多选、批量改总线、批量重命名、批量删除，以及基于当前过滤词的事件搜索定位。

7.3 右上 Property Editor

默认交互应优先服务休闲游戏常见的按钮点击、结算弹窗、轻量战斗反馈和 BGM 切换场景：默认事件采用 `Random` + `OnDemand`，默认总线优先使用 `BGM`、`SFX`、`UI`，只有在选择 `Combo` 播放模式后才展开连击相关参数。

必须支持以下属性组：

- General：Event ID、Display Name、Bus、Notes。
- Playback：Play Mode、Avoid Immediate Repeat、Loop、Fade In、Fade Out。
- Gain & Pitch：Volume、VolumeRand、Pitch、PitchRand。
- Trigger Control：Max Instances、Cooldown、Steal Policy。
- Combo：ComboPitchStep、ComboResetTime、ComboMaxStep，仅在 `Combo` 模式下展示。
- Resource：一期固定 OnDemand，标签信息。

基础范围要求：

- Volume：-96.0 dB 到 12.0 dB。
- Pitch：-2400 到 2400 cents。
- Cooldown：0 到 60 秒。
- Max Instances：0 到 999，0 表示无限制。
- Combo Max Step：0 到 999，0 表示不设上限。
- Weight：1 到 100，表示相对权重，不是百分比，也不要求总和为 100。
- TrimStartMs、TrimEndMs：最小 0 ms；编辑器会按源文件实际时长动态限制，校验阶段也会检查是否超出源文件长度。
- FadeInMs、FadeOutMs：自 AudioForge 0.04 起新增；编辑器提供波形手柄和数值框双向编辑，导出时会直接烘焙进运行时音频文件。
- LoopStartMs、LoopEndMs：自 AudioForge 0.04 起已开放编辑器侧可视化精修与时间轴标注，但当前仍不参与试听渲染和现阶段运行时执行。

7.4 右下 Contents Editor

组件建议：QTableView + 自定义 TableModel。若后续需支持内联校验、排序与拖拽重排，不建议使用 QTableWidget。

列定义：

- Clip ID
- Source Path
- Export Path
- Weight
- Start Offset
- End Offset
- 标签/备注

能力要求：

- 支持从操作系统拖入 .wav、.mp3、.ogg。
- 支持多文件批量导入。
- 支持文件存在性校验。
- 支持权重批量编辑。
- 支持导出路径预览。
- 自 AudioForge 0.04 起，片段编排页新增波形编辑台，支持直接拖拽裁剪起点/终点、淡入/淡出手柄、循环区手柄、滚轮缩放、双击聚焦选区，以及通过游标快速把当前时间写入起点、终点或循环区。

8. 领域模型设计

8.1 核心实体

Folder

- id：编辑期唯一标识。
- name：显示名称。
- children：Folder 或 Event 节点。

Event

- id：运行时唯一标识，需全局唯一。
- display_name：编辑器显示名，可为空。
- bus：所属总线。
- play_mode：Random、Sequence、Combo。
- avoid_immediate_repeat：是否避免连续命中同一 Clip。
- volume_db：基础音量。
- volume_rand_min_db、volume_rand_max_db：音量随机范围。
- pitch_cents：基础音高。
- pitch_rand_min_cents、pitch_rand_max_cents：音高随机范围。
- max_instances：并发实例数限制。
- cooldown_seconds：事件冷却时间。
- steal_policy：ReachedLimit 时的处理策略。
- combo_pitch_step_cents：Combo 每步附加音高。
- combo_reset_seconds：Combo 重置时间。
- combo_max_step：Combo 最大累计步数。
- load_policy：资源加载策略。
- clips：Clip 列表。
- notes：备注。

Clip

- id：事件内唯一标识，建议全局也唯一。
- source_path：源文件绝对路径或工程相对路径。
- export_path：导出资源目标相对路径。
- asset_key：运行时资源键。
- weight：权重。
- trim_start_ms：裁剪起点。
- trim_end_ms：裁剪终点。
- tags：标签列表。

8.2 命名规范

- Event ID 必须全局唯一。
- Event ID 只允许字母、数字和下划线，且必须以字母开头。
- 推荐格式：系统_功能_语义，例如 UI_Click_Normal、BGM_Lobby_Main、SFX_Hit_Light。
- Display Name 可使用中文，仅用于编辑器显示。
- Bus 名称必须来自受控枚举，不允许自由文本。

9. 行为语义定义

9.1 Random

- 在当前 Event 的 Clips 集合中按权重进行离散随机选择。
- Weight 的含义是未归一化权重，而非百分比。
- 若 avoid_immediate_repeat 为 true 且 Clip 数量大于 1，则本次候选集合需排除上次命中的 Clip。
- 若排除后集合为空，则回退为原始全集合。

9.2 Sequence

- Clips 按显示顺序形成序列。
- 每次触发时取当前游标对应 Clip，播放完成后或触发成功后将游标前进 1。推荐以“触发成功后前进”为标准。
- 游标达到尾部后回到 0。
- Sequence 游标默认按 Event 全局维护，而非按调用者实例维护。

9.3 Combo

- Combo 基于 Event 级状态机，不基于 Clip 状态机。
- 连续两次成功触发间隔小于等于 combo_reset_seconds 时，combo step 加 1；否则重置为 0。
- 实际播放音高 = 基础 Pitch + 随机 Pitch + step * combo_pitch_step_cents。
- 当 combo_max_step 大于 0 时，累计 step 不得超过该上限。
- Combo 模式下 Clip 选择默认仍按 Random 语义执行，除非后续增加 ComboSequence 子模式。

9.4 Max Instances

- 统计对象为同一 Event 正在播放中的实例数。
- 0 表示无限制。
- 当达到上限时，由 steal_policy 决定处理方式。

9.5 Steal Policy

- RejectNew：拒绝新播放请求。
- StopOldest：终止最早开始的实例并播放新实例。
- StopQuietest：预留策略，第一期不开放界面编辑，当前试听和 Unity 验证运行时均未实现。

9.6 Cooldown

- 以 Event 为粒度记录最后一次成功触发时间。
- 若本次触发时间与最后成功触发时间间隔小于 cooldown_seconds，则直接拒绝，不进入选 Clip 阶段。

11.3 字段契约

顶层字段：

- SchemaVersion：整数，必填。运行时据此决定兼容策略。
- GeneratedAt：ISO 8601 时间戳，必填。
- ProjectName：字符串，必填。
- Buses：字符串数组，必填。
- BusConfigs：总线对象数组，可选；若存在则应作为总线初始化状态的权威来源。
- Events：字典，键为 Event ID，必填。

BusConfig 字段：

- Name：字符串，必填。
- ParentBus：字符串，可选，默认 Master。
- VolumeDb：浮点数，可选，默认 0.0。
- IsMuted：布尔，可选，默认 false。

Event 字段：

- Bus：字符串，必填，必须存在于 Buses 中。
- PlayMode：字符串，必填，可选值为 Random、Sequence、Combo。
- AvoidImmediateRepeat：布尔，可选，默认 false。
- VolumeDb：浮点数，可选，默认 0.0。
- VolumeRandDb：长度为 2 的数组，可选，默认 [0.0, 0.0]。
- PitchCents：整数，可选，默认 0。
- PitchRandCents：长度为 2 的数组，可选，默认 [0, 0]。
- MaxInstances：整数，可选，默认 0。
- CooldownSeconds：浮点数，可选，默认 0.0。
- StealPolicy：字符串，可选，默认 RejectNew。
- LoadPolicy：字符串，可选，默认 OnDemand；一期工具与 Unity 验证运行时固定按 OnDemand 工作。
- ComboPitchStepCents：整数，仅 Combo 模式使用。
- ComboResetSeconds：浮点数，仅 Combo 模式使用。
- ComboMaxStep：整数，可选，默认 0，表示无限累计。
- Clips：数组，必填，长度至少为 1。

Clip 字段：

- ClipId：字符串，必填。
- AssetKey：字符串，必填。Unity 运行时据此加载资源。
- Weight：整数，必填，范围 1 到 100，表示相对权重而非百分比。

11.4 版本兼容规则

- 运行时必须显式检查 SchemaVersion。
- 若版本高于当前支持上限，运行时应报错并拒绝继续初始化。
- 若版本低于当前版本但仍在兼容范围内，可走兼容适配器。
- 任何破坏性变更都必须升级 SchemaVersion。

12. 资源导出规范

12.1 资源身份

资源必须同时具备以下三个层次的身份：

- ClipId：逻辑片段标识。
- AssetKey：运行时加载键。
- SourcePath：编辑期源文件路径。

SourcePath 可以变，AssetKey 应保持稳定；AssetKey 可以映射到不同物理打包系统；ClipId 则用于事件内部管理和调试。

12.2 导出目录建议

```text
Export/
  AudioData.json
  AudioEventID.cs
  AudioManifest.json
  Assets/
    ui/
    battle/
    bgm/
```

12.3 AudioManifest.json

建议增加资源清单，用于 Unity 或 CI 校验导出资源一致性。字段建议包括：

- AssetKey
- SourcePath
- ExportPath
- FileSize
- Hash
- BuildFingerprint
- ReferencedByEvents

自 AudioForge 0.05 起，`BuildFingerprint` 用于描述资源内容级构建指纹，默认至少覆盖源文件 Hash、运行时格式、Trim、Fade 和 Loop 边界。它的用途是支持增量构建时安全复用旧导出资源，而不是改变 Unity 运行时消费方式。

12.4 资源复制规则

- 仅复制被 Event 引用到的资源。
- 相同 AssetKey 对应不同源文件时，视为 Error。
- 相同源文件被多个 Event 引用时，只导出一份物理文件。
- 默认覆盖同名目标文件，但需优先比对 `BuildFingerprint` 与导出状态，避免无意义重建。
- 自 AudioForge 0.05 起，增量构建允许从上一次成功导出结果中复用未变化资源，但整个导出仍必须先写入临时目录，再以原子替换方式提交，禁止保留半成品覆盖旧导出。

13. C# 代码生成规范

13.1 生成目标

为 Unity 端生成强类型事件入口，降低魔法字符串使用。

13.2 建议产物

- AudioEventID.cs：事件枚举。
- AudioEventNames.cs：枚举到字符串的映射辅助类，可选。

13.3 枚举生成规则

- 枚举成员来自全部 Event ID。
- 枚举生成顺序必须稳定，建议按 Event ID 字典序排序。
- 非法 C# 标识符需在校验阶段直接报错，不建议静默替换。

示例：

```csharp
public enum AudioEventID
{
    BTL_Combo_Hit,
    UI_Click_Normal
}
```

14. Build 管线设计

14.1 Build 阶段

Build 必须按以下顺序执行：

1. 保存内存数据快照。
2. 执行静态校验。
3. 解析构建范围（全量 / 增量 / 选中）。
4. 基于当前工程与上次成功导出结果生成 BuildPlan。
5. 生成运行时数据模型。
6. 生成 AudioData.json。
7. 生成代码文件。
8. 生成资源清单。
9. 按 BuildPlan 重建或复用音频资源。
10. 输出 Build 报告。

14.2 失败策略

- 若存在 Error 级问题，Build 必须终止，不允许导出不完整产物。
- 若仅存在 Warning，允许继续导出，但 Build 报告需明确列出。
- 任一步骤失败时，禁止保留部分新产物覆盖旧产物。建议写入临时目录并在成功后原子替换。

14.3 Build 报告

建议额外输出 BuildReport.json，包含：

- BuildId
- StartedAt
- FinishedAt
- ProjectVersion
- SchemaVersion
- EventCount
- ClipCount
- WarningCount
- ErrorCount
- ExportedFiles
- BuildPlan.RequestedScope
- BuildPlan.EffectiveScope
- BuildPlan.SelectionLabel
- BuildPlan.RebuiltAssetKeys
- BuildPlan.ReusedAssetKeys
- BuildPlan.RemovedAssetKeys
- BuildPlan.OutOfScopeDirtyAssetKeys

15. 数据校验规范

15.1 级别定义

- Error：阻止保存或阻止 Build。
- Warning：允许保存与 Build，但需要用户确认风险。
- Info：仅提示，不阻断流程。

15.2 必须实现的 Error 规则

- Event ID 为空。
- Event ID 重复。
- Event ID 含非法字符。
- Event 无 Clip。
- Clip 源文件不存在。
- Weight 小于 1。
- Bus 不在受控枚举内。
- VolumeRand 最小值大于最大值。
- PitchRand 最小值大于最大值。
- Combo 模式缺失 ComboPitchStepCents 或 ComboResetSeconds。
- 同一 AssetKey 指向多个不同源文件。
- 导出目录不可写。

15.3 必须实现的 Warning 规则

- Event 只有一个 Clip 但设置了 AvoidImmediateRepeat。
- Sequence 模式只有一个 Clip。
- Cooldown 大于 5 秒。
- MaxInstances 为 1 且 StealPolicy 非 RejectNew。
- 资源格式不是推荐格式。
- 导出资源路径存在大小写冲突风险。

15.4 即时校验与 Build 校验

- 即时校验：用户编辑字段时尽快反馈本地规则。
- Build 校验：在导出前重新做全量校验，保证最终产物可信。

16. 本地试听规范

16.1 目标

本地试听用于尽量还原 Unity 运行时的事件选择、参数计算与实例生命周期，作为策划与音频设计阶段的听感校对工具。

16.2 一致性要求

试听必须与运行时在以下逻辑上完全一致：

- Random 权重选择
- AvoidImmediateRepeat 规则
- Sequence 游标推进
- Combo step 累加与重置
- Cooldown 拒绝逻辑
- Max Instances 判定策略
- StopOldest 抢占策略
- 基于 Trim 的真实试听时长
- Master 与事件输出总线的预览音量、静音叠加
- 属性编辑器中所有音高相关参数在本地试听里默认按保时长变调处理
- Combo 连击步进按半音档位累加

16.3 可接受差异

以下差异允许存在：

- 音频解码库不同导致的波形细节差异
- Unity 音频导入设置导致的压缩或延迟差异
- 不同平台硬件缓冲带来的播放起始偏差
- 当前工具端仍未复刻 Unity 原生总线 DSP、效果器链与空间化处理
- 当前工具端已将基础音高、随机音高和 Combo 附加音高统一为保时长试听；Unity 运行时若仍直接使用 `AudioSource.pitch`，则会与工具端在片段时长变化上存在已知差异。

- 当前工具端已支持 Combo 连击加成的保时长变调试听；Unity 运行时若要与该听感完全一致，需后续接入同等的变调 DSP 或自定义保时长变调实现。

16.3.1 总线预览状态说明

- 工具端的“试听总线”仅用于本地试听与响度监视，不写入 .afproj 导出配置，也不参与 AudioData.json 生成。
- 试听总线包含一个隐式 Master 和工程中定义的事件总线；最终有效音量按 Master 与事件总线线性乘算后，再叠加到事件音量上。

16.4 试听技术选型

- 第一优先：pygame.mixer，适合快速播放与原型阶段。
- 备选：pydub + simpleaudio，适合更灵活的片段处理。
- 不建议将试听实现和导出实现混写在同一 Controller 中，应提取独立 PreviewService。

17. Unity 运行时接口建议

运行时最小接口建议如下：

```csharp
public interface IAudioRuntime
{
    bool PlayEvent(string eventId);
    bool PlayEvent(AudioEventID eventId);
    void StopEvent(string eventId);
    void StopBus(string busName);
    void SetBusVolume(string busName, float volumeDb);
    bool HasEvent(string eventId);
    void Preload(string eventId);
    void Unload(string eventId);
}
```

实现约束：

- PlayEvent 的返回值表示是否成功接受播放请求，而不是是否已经播放完成。
- HasEvent 仅检查数据库中是否存在，不应触发加载。
- Preload 与 Unload 是资源层能力，不应改写事件数据本身。

18. 性能目标

以下目标适用于桌面开发环境第一期实现：

- 1000 个 Event、5000 个 Clip 的工程在 3 秒内完成打开。
- 全量校验在 2 秒内完成。
- 生成 AudioData.json 与代码文件在 2 秒内完成，不含资源复制。
- 左侧树展开和筛选在常规编辑操作下无明显卡顿。

若第一期无法完全满足，应在实现中至少保留异步校验、分段刷新和延迟加载的扩展点。

19. 可用性要求

- 支持 Undo/Redo。
- 支持自动保存或崩溃恢复。
- 支持脏状态提示。
- 支持最近工程列表。
- 支持搜索与过滤。
- 支持批量修改 Clip 权重。
- 支持复制 Event 并自动建议新 ID。

一个可交付的 AudioForge 版本，至少需要满足以下条件：

25. 结论

AudioForge 的核心价值不在于“做出一个像 Wwise 的界面”，而在于建立一条稳定的数据驱动音频生产链路。只要工具、导出数据和 Unity 运行时三者之间的职责边界、字段契约和行为语义足够清晰，这套方案就能长期维护，并且不会反向依赖引擎侧实现细节。
